# QA Checklist — no-one-alone 제안서 제출 전 점검
> Created: 2026-05-08 01:39
> Last Updated: 2026-05-20 13:00
> Backlog: P-07, D-03, T-04, M-01, M-02, M-03, M-05, M-06, Q-01, Q-02, Q-04

> 이 체크리스트는 2026-05-31 온라인 모집 마감 전, 공모 제안서 제출물을 검수하기 위한 문서다.
> 현재 단계에서는 구현물이 없으므로 앱 실행 QA가 아니라 제안서 품질과 심사 기준 대응을 점검한다.

## 1. 양식 준수

- [ ] PPTX 표지에 팀명, 프로젝트명, 제출일 입력
- [ ] 제안 요약서 표 구조 유지
- [ ] 제안 요약서 최대 2p 이내
- [ ] 본문은 표지·요약서 제외 10p 이내
- [ ] 각 페이지 주제에 맞게 구체적으로 작성
- [ ] 텍스트 나열을 줄이고 표·도식·흐름도로 설명
- [ ] 8분 발표 / 5분 Q&A 가능한 분량

- [ ] 제안내용의 목적 및 필요성 포함
- [ ] 제안내용의 상세 설명 포함
- [ ] 제안내용의 차별성 포함
- [ ] 기대효과 포함
- [ ] 제안내용 관련 자료 포함 여부 판단
- [ ] 기타 자유 기재 필요 여부 판단

## 4. Track 2 및 가점 조건

- [ ] Track 2 선택 이유가 명확함
- [ ] 현재 구현물이 없고 MVP 개발 단계(7/1~9/21)에서 구현한다는 점이 일관됨
- [ ] 모바일 신분증 활용 지점 2개(가입·로그인, 집행 직전)가 자연스러움
- [ ] 개인 자격 확인과 공공 마이데이터 연계가 MVP 제외 범위로 분리됨
- [ ] Open DID 활용이 지원 확인 VC 발급·VP 검증에 연결됨
- [ ] OmniOne Chain 활용이 지원 확인 VC 메타데이터·지원 완료 증명에 연결됨
- [ ] Open DID + OmniOne Chain 가점 +10% 확보가 MVP 필수 성공 기준으로 명시됨
- [ ] 온체인에는 DID 원문이 아니라 해시·집계 데이터만 기록한다고 명시됨

## 5. 내용 정합성

- [ ] “이미 구현됨”, “배포 완료”처럼 현재 상태와 맞지 않는 표현 없음
- [ ] 문제 수치와 출처가 제안서 본문 또는 관련 자료에 연결됨
- [ ] 지자체 B2G SaaS와 보험회사 PoC·리스크 모델링 수익 구조가 서로 충돌하지 않음
- [ ] 개인정보·민감정보 처리 리스크에 대한 방어 문장이 있음
- [ ] MVP 구현 범위와 이후 사업 확장 범위가 구분됨
- [ ] 보험사 데이터 API가 MVP 구현 대상이 아니라 파일럿 이후 확장 후보로 표시됨

---

## 6. MVP 개발 단계 계획 점검

- [ ] 7/1~9/21 구현 일정이 주차별로 정리됨
- [ ] MVP 포함 범위와 제외 범위가 구분됨
- [ ] 핵심 시연 플로우가 하나로 연결됨: OmniOne CX 가입·로그인 → 솔이 감지 → 담당자 승인 → 집행 직전 본인 재확인 → VC/txId 표시
- [ ] 복지 매칭은 핵심 감지 기능이 아니라 지원 연결 보조 기능으로 표현됨
- [ ] 모바일 신분증, Open DID, OmniOne Chain 활용이 각각 시연 지점과 연결됨
- [ ] 최소 1개 실제 VC 발급, VP 검증, txId 표시 성공을 제출 목표로 명시함
- [ ] 한국사회보장정보원 복지서비스 API는 실연동하고, 개인 자격 확인은 파일럿 이후로 분리한다고 명시됨
- [ ] SDK 연동 지연, AI 품질, 개인정보, 네트워크 장애 대응 계획이 있음
- [ ] 실사용자 데이터 없이 데모 데이터로 시연한다는 방침이 명확함
- [ ] 실제 주소·상세 상황 설명은 MVP 데모에서 저장하지 않고, 지역 코드·해시·데모 데이터 중심으로 시연한다는 방침이 명확함
