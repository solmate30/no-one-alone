# QA Checklist — no-one-alone 제안서 제출 전 점검
> Created: 2026-05-08 01:39
> Last Updated: 2026-05-08 17:35
> Backlog: P-07, D-03, M-01, M-02, M-03, M-05, Q-01, Q-02, Q-04

> 이 체크리스트는 2026-05-31 온라인 모집 마감 전, 공모 제안서 제출물을 검수하기 위한 문서다.
> 현재 단계에서는 구현물이 없으므로 앱 실행 QA가 아니라 제안서 품질과 심사 기준 대응을 점검한다.

## 1. 양식 준수

- [ ] PPTX 표지에 팀명, 프로젝트명, 제출일 입력
- [ ] 제안 요약서 표 구조 유지
- [ ] 제안 요약서 최대 2p 이내
- [ ] 본문은 표지·요약서 제외 10p 이내
- [ ] 각 페이지 주제에 맞게 구체적으로 작성
- [ ] 텍스트 나열을 줄이고 표·도식·흐름도로 설명
- [ ] 8분 발표 / 5분 Q&A 가능한 분량

---

## 2. 필수 목차 충족

- [ ] 제안내용의 목적 및 필요성 포함
- [ ] 제안내용의 상세 설명 포함
- [ ] 제안내용의 차별성 포함
- [ ] 기대효과 포함
- [ ] 제안내용 관련 자료 포함 여부 판단
- [ ] 기타 자유 기재 필요 여부 판단

---

## 3. 심사 기준 대응

| 심사 항목 | 배점 | 점검 질문 | 상태 |
|:---|:---:|:---|:---:|
| 창의성 | 35 | 기존 복지 시스템과 다른 조합이 명확한가? | [ ] |
| 실현 가능성 | 35 | 결선 기간 안에 구현 가능한 범위로 좁혔는가? | [ ] |
| 사업성 | 30 | 수익 구조와 확산 경로가 과장 없이 설명되는가? | [ ] |

---

## 4. Track 2 및 가점 조건

- [ ] Track 2 선택 이유가 명확함
- [ ] 현재 구현물이 없고 결선에서 MVP를 개발한다는 점이 일관됨
- [ ] 모바일 신분증 활용 지점 3개가 자연스러움
- [ ] Open DID 활용이 제보자·관계 인증에 연결됨
- [ ] OmniOne Chain 활용이 집행 증명·CSR 투명성에 연결됨
- [ ] 온체인에는 DID 원문이 아니라 해시·집계 데이터만 기록한다고 명시됨

---

## 5. 내용 정합성

- [ ] “이미 구현됨”, “배포 완료”처럼 현재 상태와 맞지 않는 표현 없음
- [ ] 문제 수치와 출처가 제안서 본문 또는 관련 자료에 연결됨
- [ ] B2G, 보험, CSR, 데이터 API 수익 구조가 서로 충돌하지 않음
- [ ] 개인정보·민감정보 처리 리스크에 대한 방어 문장이 있음
- [ ] 결선 구현 범위와 이후 사업 확장 범위가 구분됨

---

## 6. 결선 MVP 계획 점검

- [ ] 7/1~9/21 구현 일정이 주차별로 정리됨
- [ ] 결선 MVP 포함 범위와 제외 범위가 구분됨
- [ ] 핵심 시연 플로우가 하나로 연결됨: 이웃 제보 → Soli 스코어 → 담당자 승인 → 온체인 기록
- [ ] 모바일 신분증, Open DID, OmniOne Chain 활용이 각각 시연 지점과 연결됨
- [ ] SDK 연동 지연, AI 품질, 개인정보, 네트워크 장애 대응 계획이 있음
- [ ] 실사용자 데이터 없이 데모 데이터로 시연한다는 방침이 명확함

---

## 7. 제출 전 최종 확인

- [ ] 파일명 형식 확인: `TrackNo_팀명_프로젝트명_yymmdd.pptx`
- [ ] 맞춤법과 숫자 단위 확인
- [ ] 팀원 정보 확인
- [ ] 발표 예상 질문 10개 작성
- [ ] 최종 제출 파일을 별도 백업
