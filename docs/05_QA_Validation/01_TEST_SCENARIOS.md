## 1. 검증 기준 — 제안서 심사 관점

현재 단계는 구현 전 설계 단계이므로, 앱 기능 테스트가 아니라 제안서가 심사 기준을 설득하는지 검증한다. PPTX 양식 기준 예선 심사 항목은 창의성 35점, 실현 가능성 35점, 사업성 30점이다.

| 심사 항목 | 배점 | 핵심 질문 |
|:---|:---:|:---|
| 창의성 | 35 | 기존 복지 시스템·보험·AI 챗봇과 다른 조합인가? |
| 실현 가능성 | 35 | 결선 기간 내 구현 가능한 MVP 범위인가? |
| 사업성 | 30 | 공공·보험·CSR 시장에서 지속 가능한 수익 구조인가? |

---

## 2. 창의성 검증

### C-01: 기존 방식 대비 새로움

| 확인 항목 | 제안서 증거 | 합격 기준 |
|:---|:---|:---|
| 복지 사각지대 발굴 | 이웃 제보 + 가족 케어 + 자가 신청 파이프라인 | 데이터 밖 사람을 찾는 구조가 설명됨 |
| AI 활용 | Soli 대화 기반 위기 키워드·응답 패턴 분석 | 단순 챗봇이 아닌 선제 감지 역할이 명확함 |
| 블록체인 활용 | OmniOne Chain 집행 기록 | 억지 활용이 아니라 투명성 문제와 연결됨 |
| 모바일 신분증 활용 | 제보자·대상자·수급자 확인 | 허위 신고·자격 검증·부정 수급 방지와 연결됨 |

## 3. 실현 가능성 검증

### F-01: 결선 MVP 범위

| 구현 후보 | 결선 포함 여부 | 검증 질문 |
|:---|:---:|:---|
| Soli 챗봇 위기 감지 | 포함 | 2026-07-01~2026-09-21 안에 최소 기능 구현 가능한가? |
| 이웃 제보 플로우 | 포함 | 모바일 신분증 인증과 연결 가능한가? |
| 담당자 대시보드 | 포함 | 우선순위 리스트와 상세 요약만으로 시연 가능한가? |
| OmniOne Chain 기록 | 포함 | 최소 1개 TX 기록으로 가점 요건을 보여줄 수 있는가? |
| 실제 행정 시스템 연동 | 제외 | 결선 이후 파일럿 단계로 분리했는가? |
| 보험사 실 API 계약 | 제외 | 사업 모델로만 제시했는가? |

### F-02: 결선 구현 일정 검증

| 검증 항목 | 합격 기준 |
|:---|:---|
| 7/1~9/21 일정 분해 | 주차별 산출물이 명확함 |
| 핵심 시연 플로우 | 이웃 제보 → Soli 스코어 → 담당자 승인 → 온체인 기록으로 이어짐 |
| 가점 조건 | Open DID와 OmniOne Chain을 각각 최소 1회 이상 시연할 계획이 있음 |
| 기술 리스크 | SDK 지연, AI 품질, 개인정보, 네트워크 장애 대응이 있음 |
| 범위 통제 | 가족 케어·자가 신청·보험 API는 결선 핵심 시연에서 제외하거나 보조로 둠 |

## 6. 결선 MVP QA 시나리오 (진출 후 적용)

아래 항목은 5월 제안서 제출 시 실행하지 않는다. 결선 진출 후 MVP 구현이 시작되면 실제 QA 기준으로 전환한다.

| 시나리오 | 성공 기준 |
|:---|:---|
| 이웃 제보 E2E | 주소·상황 입력 → 모바일 신분증 인증 → 제보 접수 완료 |
| Soli 위기 감지 | 위험 문장 입력 시 위기도 스코어 7 이상 산출 |
| 담당자 우선순위 | 제보·스코어가 높은 케이스가 리스트 상단에 표시 |
| 지원 승인 | 담당자가 지원 종류를 선택하고 승인 상태로 변경 |
| 온체인 기록 | 제보 또는 집행 이벤트의 TX 해시가 화면에 표시 |
| 개인정보 보호 | DID 원문과 대화 원문이 화면·온체인에 노출되지 않음 |
