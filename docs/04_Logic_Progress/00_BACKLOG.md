## 0. 현재 상태

현재 no-one-alone은 **구현 전 설계 단계**다. 2026년 5월의 목표는 동작하는 MVP가 아니라, `refs/TrackNo_팀명_프로젝트명_yymmdd.pptx` 양식에 맞춘 공모 제안서 제출이다.

다만 Track 2는 MVP 모델 개발·시연을 요구하므로, 제안서에는 **5월 제출 계획**과 **MVP 개발 단계 7/1~9/21 구현 계획**을 함께 담는다.

## 4. 제안서에 포함할 MVP 약속

아래 항목은 5월에 구현하지 않는다. 대신 제안서에서 “MVP 개발 단계에서 실제로 만들 MVP 범위”로 명확히 제시한다.

### 4-1. MVP 포함 범위

- [ ] Soli 챗봇: 기본 대화, 위기 키워드 감지, 위기도 스코어 출력
- [ ] 시민 제보 플로우: 주소·상황 입력, 제보자 모바일 신분증 인증, 제보 접수 완료
- [ ] 공공 마이데이터 자격 확인 흐름: 동의 화면, Mock 조회 결과, 담당자 확인 배지
- [ ] 담당자 대시보드: 위기 우선순위 리스트, 대상자 상세, Soli 요약, 지원 승인 버튼
- [ ] OmniOne Chain 기록: 제보 또는 지원 집행 중 최소 1개 이벤트 TX 기록 및 해시 표시
- [ ] Open DID/VC: 제보자 인증 중 최소 1개 신원 증명 플로우 구현

### 4-2. MVP 제외 범위

- [ ] 실제 행정 시스템 연동 제외
- [ ] 실제 공공 마이데이터 운영 API 계약·심사 완료 제외
- [ ] 실제 보험사 API 계약·연동 제외
- [ ] 실제 복지금 지급·정산 제외
- [ ] 실사용자 개인정보 수집 제외 (데모 데이터 사용)
- [ ] 정식 서비스 수준의 운영/장애 대응 제외

## 6. MVP 리스크와 대응

| 리스크 | 대응 |
|:---|:---|
| OmniOne CX/Chain SDK 연동 지연 | 데모 모드와 실제 연동 모드를 분리하고, 최소 1개 인증·TX 성공을 목표로 축소 |
| 공공 마이데이터 운영 API 연동 지연 | 결선 MVP에서는 동의 화면과 Mock 응답으로 자격 확인 흐름을 시연하고, 실제 운영 API 계약·심사는 파일럿 단계로 분리 |
| AI 응답 품질 불안정 | 위기 키워드 룰 기반 스코어링을 fallback으로 구현 |
| 개인정보 이슈 | 실사용자 데이터 없이 데모 데이터와 DID 해시만 사용. 마이데이터 원문은 저장하지 않고 확인 결과와 이벤트 해시만 남김 |
| 범위 과대 | Phase 2 가족 안부 케어는 제외하고, 핵심 시연은 이웃 제보→담당자 승인→온체인 기록에 집중 |
| 발표 당일 네트워크 장애 | 로컬 데모 데이터와 녹화 영상을 백업으로 준비 |
