## 3. 제안내용의 목적 및 필요성

**One-line message**: 시스템이 찾지 못하는 사람들이 있고, 그들은 죽어서야 발견된다.

### 핵심 메시지

복지 사각지대의 문제는 지원 제도가 없어서가 아니라, 위기 가구를 제때 찾지 못해서 발생한다.

### 문제 현황

| 지표 | 수치 | 의미 |
|:---|:---:|:---|
| 전국 복지대상자 | 1,488.5만 명(2023) | 2019년 1,153.1만 명 대비 335.4만 명 증가 |
| 복지 담당자 관리 부담 | 1인 평균 1,027명(2020 보도 기준) | 수동 발굴에 구조적 한계가 있음 |
| 발굴 후 실제 지원 | 누적 666만 명 중 290만 명(약 44%) | 발굴 이후에도 절반 이상이 지원으로 연결되지 않음 |
| 위기정보 수집 주기 | 격월(연 6회) | 상시 위기 감지가 어려움 |

출처: 보건복지부 2024 복지 사각지대 발굴 보도자료, 보건복지부 제출 통계 기반 보도(뉴시스, 2024), 한국일보 2020 보도

### 사각지대가 생기는 이유

- 주소지와 실거주지가 다른 고시원·쪽방·여관 거주자는 행정 데이터에 잘 잡히지 않는다.
- 정신질환, 인지저하, 사회적 고립 상태의 대상자는 본인이 직접 신청하기 어렵다.
- 1인 가구 증가로 주변 이웃이 이상 징후를 알아차려도 신고 채널이 명확하지 않다.
- 기존 고독사 보험은 위험 대상자를 알 수 없어 판매와 리스크 관리 모두 어렵다.

## 4. 제안내용의 상세 설명

**One-line message**: 국가가 못 찾는 사람을, 이웃이 찾고, AI가 연결하고, 블록체인이 증명한다.

### 서비스 정의

no-one-alone은 "발굴 → 판단 → 집행 → 기록"을 하나로 연결하는 복지 위기 대응 플랫폼이다.

```text
[발굴] 현장 접점 제보 / 자가 신청 / 담당자 등록
  -> [판단 보조] Soli AI 위기 신호 요약 + 모바일 신분증 확인 + 마이데이터 자격 확인 흐름
  -> [집행] 담당자 대시보드 우선순위 확인 + 지원 승인
  -> [기록] OmniOne Chain에 집행 증명 해시 기록
```

### 핵심 기능

| 기능 | 대상 | 설명 |
|:---|:---|:---|
| 이웃·현장 접점 제보 | 시민·고시원 관리자·생활지원사·통장/반장 | 주소와 상황을 입력하고 모바일 신분증으로 제보자 신원을 확인한다. |
| 자가 복지 신청 | 위기 대상자 | Soli와 대화하며 받을 수 있는 복지를 추천받는다. MVP 개발 단계에서는 공공 마이데이터 동의 화면과 Mock 검증 결과로 자격 확인 흐름을 시연한다. |
| Soli AI | 대상자·담당자 | 독거노인·취약계층에게 정기 안부를 묻고, 감정 표현·위기 키워드·응답 공백을 기반으로 위기 신호를 요약한다. 최종 판단은 담당자가 한다. |
| 담당자 대시보드 | 복지 담당자 | 오늘 우선 확인이 필요한 사람을 스코어 순으로 확인한다. |
| 온체인 기록 | 지자체·기업 | 제보·판단·집행의 증명 해시를 기록해 투명성을 높인다. |

### 초기 유입 경로

| 경로 | 첫 신호 | 운영 방식 |
|:---|:---|:---|
| 현장 접점 제보 | 고시원·쪽방·편의점·통장/반장 등 지역 접점의 이상 징후 | 지자체 캠페인과 협력 기관을 통해 제보 채널을 배포한다. |
| 자가 신청 | 본인이 복지 정보나 긴급 도움을 찾는 대화 | Soli가 받을 수 있는 복지를 안내하고 담당자 검토로 연결한다. |
| 담당자 등록 | 독거노인·취약계층 등 기존 관리 대상자의 정기 안부 확인 필요 | 대상자 본인 동의 후 Soli 정기 안부 대화를 연결하고 대시보드에서 추적한다. |

### 모바일 신분증 활용 지점

| 사용 시점 | 역할 | 효과 |
|:---|:---|:---|
| 이웃 제보 시 | 제보자 본인 확인 | 허위 신고 방지 |
| 복지 자격 확인 시 | 대상자 본인 확인 | 자격 검증 간소화 |
| 지원 집행 직전 | 수급자 본인 확인 | 부정 수급 방지 |

### 공공 마이데이터 시연 지점

| 사용 시점 | MVP 시연 방식 | 효과 |
|:---|:---|:---|
| 복지 자격 확인 | 대상자 동의 화면 → Mock 자격 정보 조회 → 담당자 대시보드 "자격 확인 완료" 배지 표시 | 서류 요청 없이 신청-확인 흐름을 보여준다. |
| 담당자 검토 | 대시보드 리스트에는 배지만 표시. 케이스 상세 화면 진입 후 건강보험 자격득실·가구원 구성 등 보조 정보 열람 가능 | 개인정보 최소 노출 원칙 유지. 최종 판단은 담당자가 한다. |
| 증명 기록 | 자격 확인 이벤트 해시를 온체인 기록 후보로 표시 | 원문을 노출하지 않고 확인 이력만 증명한다. |

### OmniOne Chain 활용 구조

no-one-alone은 복지 데이터 원문을 블록체인에 올리지 않는다. 대신 제보·동의·승인·집행 이벤트의 해시만 OmniOne Chain에 기록해, 지자체·후원 기업·감사 주체가 같은 기록을 기준으로 사후 검증할 수 있게 한다.

| 이벤트 | 온체인 기록 | 검증 가치 |
|:---|:---|:---|
| 제보 접수 | 제보자 DID 해시 + 상황 해시 + 타임스탬프 | 허위 제보 억제, 제보 접수 시점 증명 |
| 자격 확인 | 대상자 동의 이벤트 해시 + 자격 확인 결과 해시 | 원문 노출 없이 동의·확인 이력 증명 |
| 담당자 승인 | 승인자 해시 + 케이스 해시 + 승인 시각 | 누가 어떤 근거로 승인했는지 사후 검증 |
| 지원 집행 | 수급자 DID 해시 + 지원 종류·금액 해시 + TX 해시 | 복지 집행과 CSR 후원 집행 증빙 비용 절감 |

**개인정보 경계**: 주소, 상황 설명, Soli 대화 원문, 마이데이터 원문은 DB에 암호화 저장하고, OmniOne Chain에는 DID 해시·이벤트 해시·집행 증명 해시만 기록한다.

### 판단 책임 원칙

- AI는 수급 여부나 지원 여부를 자동 결정하지 않는다.
- Soli는 위기 키워드, 응답 공백, 제보 이력 등 담당자가 확인할 근거를 요약한다.
- 담당자 대시보드는 "왜 고위험인지"를 함께 표시하고, 최종 승인과 집행은 사람이 수행한다.
- MVP 개발 단계에서는 위험 스코어보다 설명 가능한 근거 표시를 우선 구현한다.

### 개발·시연 범위

현재 구현된 웹앱, API, DB, OmniOne 연동은 없다. 5월 제안서 제출 후 개발·시연 대상으로 선정되면, 2026-07-01 ~ 2026-09-21 기간 동안 아래 범위를 MVP로 구현한다.

## 5. 제안내용의 차별성

**One-line message**: 기존 시스템이 놓치는 현장 신호를 제보와 AI 근거 요약으로 더 빨리 연결한다.

### 기존 시스템과의 차이

| 구분 | 기존 복지 시스템 | 고독사 보험 | no-one-alone |
|:---|:---|:---|:---|
| 발굴 방식 | 행정 데이터 중심, 정기 수집 | 보험 가입 이후 사후 보장 중심 | 제보 기반 상시 접수 + AI 근거 요약 |
| 사각지대 | 행정 데이터 밖 현장 신호 반영이 어려움 | 가입 전 위험 신호 파악이 어려움 | 이웃·현장 접점 제보로 보완 |
| 신청 방식 | 본인 신청 필수 | 해당 없음 | 제보·자가 신청·담당자 등록으로 먼저 신호 연결 |
| 집행 투명성 | 기관 내부 기록 중심 | 해당 없음 | 집행 증명 해시로 사후 검증 보완 |
| 담당자 업무 | 다수 대상 수동 확인 부담 | 해당 없음 | 위기도 우선순위 기반 1차 분류 |
| 데이터 축적 | 기관별로 분절 | 보험 계약 데이터 중심 | 제보-대화-처리 결과를 연결한 사회적 고립 신호 누적 |

### 기술 차별성

- 모바일 신분증을 억지로 붙이는 것이 아니라, 제보·자격·집행 신뢰 구조의 핵심으로 사용한다.
- Open DID는 제보자 신원 증명에 활용한다. MVP 개발 단계에서는 제보자 인증 1개 플로우로 연동 범위를 제한한다.
- OmniOne Chain은 개인정보 원문이 아니라 DID 해시, 데이터 해시, 집행 증명 해시만 기록한다. DB 로그를 대체하는 것이 아니라 지자체·보험회사·CSR 후원자·감사 주체가 함께 신뢰할 수 있는 사후 증명 레이어다.
- Soli AI는 단순 상담 챗봇이 아니라 복지 담당자에게 전달할 위기 신호를 정리하는 첫 접점이다.
- 공공 마이데이터는 자동 판정 장치가 아니라 대상자 본인 동의 시에만 사용한다. 대상자가 Soli에서 자가 신청할 때, 또는 담당자가 현장 방문 시 대상자가 직접 동의할 때 자격 확인 보조 정보를 받아볼 수 있다. 이를 통해 서류 보완을 위한 재방문을 줄이고 한 번 방문으로 상황 파악과 지원 승인 검토를 함께 진행할 수 있다. 동의 완료 사실은 Open DID Delegation 형태로 기록해 사후 검증 가능한 동의·확인 이력으로 남긴다. MVP 개발 단계에서는 동의-조회-확인 배지 흐름을 표준 규격 기반 Mock 응답으로 시연한다.

### 신뢰 문제별 기술 적용

| 심사위원 질문 | 적용 기술 | 답변 |
|:---|:---|:---|
| 허위 제보는 어떻게 줄이나? | 모바일 신분증 + Open DID + 안심 제보 인증서 VC | 제보자 본인을 확인해 책임을 귀속시키고, 제보 이력을 OmniOne Chain에 기록한 안심 제보 인증서 VC를 발급한다. 제보자는 반복 신원확인 없이 후속 확인에 응할 수 있어 선의의 제보 장벽도 낮춘다. |
| AI 오판과 낙인은 어떻게 막나? | Soli 근거 요약 + 담당자 최종 판단 | AI는 판정자가 아니라 우선순위 보조 도구이며, 스코어 근거를 함께 표시한다. |
| 서류 없이 복지 자격을 어떻게 확인하나? | 모바일 신분증 + 공공 마이데이터 + Open DID Delegation | 마이데이터는 대상자 본인 동의 시에만 사용한다. 담당자가 현장 방문 시 대상자가 즉시 동의하면 자격 보조 정보를 바로 확인할 수 있어 서류 보완 재방문을 줄일 수 있다. 동의 사실은 Open DID Delegation으로 기록되어 사후 검증 가능한 동의·확인 이력이 된다. |
| 블록체인이 꼭 필요한가? | OmniOne Chain 해시 기록 | 개인정보 원문이 아니라 집행 증명 해시를 기록해 다기관 감사와 CSR 집행 증빙 비용을 낮춘다. |
| 보험회사에 개인 정보가 넘어가나? | 집계·비식별 데이터 원칙 | 개인 단위 판매는 금지하고, 지역·연령대·위험 패턴 단위 통계와 공동 실증 결과만 제공한다. |

**핵심 해자 (Moat)**: 검증된 제보, Soli 대화 패턴, 담당자 처리 결과가 함께 쌓일수록 기존 공공 데이터로는 포착하기 어려운 사회적 고립 신호 데이터셋이 축적된다. 이 데이터는 지자체 선제 대응 정책과 비식별 통계 기반 공동 실증의 기반이 되어 경쟁자 진입 장벽이 된다.

### 사업화 및 확장 가능성

**One-line message**: 지자체에는 발굴·우선순위화 도구를 팔고, 보험회사는 비식별 통계 기반 공동 실증 파트너로 확장한다.

### 핵심 고객 1: 지자체 B2G SaaS

행정복지센터 담당자용 AI 우선순위화 도구를 기초지자체 단위 연간 라이선스로 공급한다.

| 단계 | 규모 | 연 매출 |
|:---|:---|:---|
| 파일럿 | 1개 기초지자체 | 5,000만원 |
| 확장 | 10개 기초지자체 | 5억 |
| 전국 | 226개 기초지자체 | 100억+ |

진입 전략: 해커톤 입상 후 고독사 예방·복지 사각지대 발굴 의지가 있는 1개 기초지자체와 PoC를 추진한다. 초기에는 무상 또는 저비용 시범사업으로 접수-확인 시간, 담당자 만족도, 허위 제보율을 측정하고, 성과지표 확보 후 협약·조달 채널을 검토한다.

### 확장 파트너: 보험회사 공동 실증

보험회사에는 개인 단위 위험 판정 데이터가 아니라, 고독사 예방형 사회안전망 상품을 고도화하기 위한 집계·비식별 위험 신호 통계와 공동 실증 결과를 제공한다.

| 단계 | 제공 가치 | 수익화 방식 |
|:---|:---|:---|
| PoC | 지역·연령대별 고립 신호, 제보 빈도, 응답 공백 패턴 검증 | PoC 용역비 또는 공동 실증 |
| 상품 고도화 | 고독사 예방형 사회안전망 상품 설계 참고 지표 제공 | 공동 연구 또는 리포트 구독 |
| 운영 고도화 | 가입자 동의 기반 안부 확인, 위험 신호 알림, 지자체 연계 | 보험사 솔루션 이용료 |

**보험사가 필요한 이유**: 기존 고독사 보험은 예방형 서비스와 지역 단위 위험 신호 데이터가 부족해 상품 확산이 제한적이다. no-one-alone이 축적하는 비식별 고립 신호와 제보 패턴은 보험사가 개인을 선별하는 용도가 아니라 예방형 상품과 공공 연계 모델을 설계하는 참고 지표로 활용된다.

## 8. 기타

### 팀 역량

**One-line message**: AI 기반 위기 분류와 DID·온체인 증명 흐름을 빠르게 구현·검증하는 팀.

| 역할 | 담당 | 강점 |
|:---|:---|:---|
| 팀장 / 풀스택 개발 | 개발 총괄 | Next.js, NestJS 기반 웹 서비스 구현, API 설계, 블록체인 연동 구조 학습 및 시연 설계 |
| AI·백엔드 개발 | 개발·시연 단계 합류 예정 | Soli 챗봇, 위기 스코어링, API 구현 |

**팀 역량 증거**:
- Next.js, NestJS 기반 웹 서비스 구현 역량
- Soli 챗봇, 담당자 대시보드, 증명 해시 표시까지 이어지는 단일 시연 플로우 설계 완료

### 주요 리스크와 대응

| 리스크 | 대응 |
|:---|:---|
| OmniOne CX/Chain SDK 연동 지연 | 제보자 인증 1개 플로우로 범위를 제한하고 데모 모드 병행 준비 |
| AI 응답 품질 불안정 | 룰 기반 위기 키워드 스코어링을 fallback으로 구현 |
| AI 오판·낙인 우려 | AI는 최종 판정자가 아니라 근거 요약 도구로 제한하고, 담당자 최종 승인 구조를 명시 |
| 마이데이터 연동·민감정보 우려 | MVP은 동의 화면과 Mock 응답으로 시연한다. 원문은 저장하지 않고 자격 확인 결과와 이벤트 해시만 남긴다. |
| 개인정보 우려 | 실사용자 데이터 없이 데모 데이터와 DID 해시만 사용. 보험회사에는 집계·비식별 데이터만 제공 |
| 제보 참여 부족 | 일반 시민 앱 확산보다 고시원·쪽방·생활지원사·통장/반장 등 현장 접점 제보망부터 확보 |
| 범위 과대 | 핵심 시연을 이웃 제보, 담당자 승인, 온체인 기록으로 집중 |
| 발표 당일 네트워크 장애 | 로컬 데모 데이터와 녹화 영상 준비 |
