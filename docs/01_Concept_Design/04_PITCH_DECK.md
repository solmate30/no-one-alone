## 3. 제안내용의 목적 및 필요성

**One-line message**: 위기 신호는 존재하지만, 현재 시스템은 너무 늦게 발견한다.

### 핵심 메시지

복지 사각지대의 문제는 지원 제도가 없어서가 아니라, 위기 가구를 제때 찾지 못해서 발생한다.

### 문제 현황

| 지표 | 수치 | 의미 |
|:---|:---:|:---|
| 전국 복지대상자 | 1,488.5만 명(2023) | 2019년 1,153.1만 명 대비 335.4만 명 증가 |
| 복지 담당자 관리 부담 | 1인 평균 1,027명(2020 보도 기준) | 수동 발굴에 구조적 한계가 있음 |
| 발굴 후 실제 지원 | 누적 666만 명 중 290만 명(약 44%) | 발굴 이후에도 절반 이상이 지원으로 연결되지 않음 |
| 위기정보 수집 주기 | 격월(연 6회) | 상시 위기 감지가 어려움 |

출처: 보건복지부 2024 복지 사각지대 발굴 보도자료, 보건복지부 제출 통계 기반 보도(뉴시스, 2024), 한국일보 2020 보도

### 사각지대가 생기는 이유

- 주소지와 실거주지가 다른 고시원·쪽방·여관 거주자는 행정 데이터에 잘 잡히지 않는다.
- 정신질환, 인지저하, 사회적 고립 상태의 대상자는 본인이 직접 신청하기 어렵다.
- 1인 가구 증가로 주변 이웃이 이상 징후를 알아차려도 신고 채널이 명확하지 않다.
- 기존 고독사 보험은 위험 대상자를 알 수 없어 판매와 리스크 관리 모두 어렵다.

## 4. 제안내용의 상세 설명

**One-line message**: 국가가 못 찾는 사람을, AI가 찾고, 복지가 닿고, 블록체인이 증명한다.

### 서비스 정의

no-one-alone은 "발굴 → 판단 → 집행 → 기록"을 하나로 연결하는 복지 위기 대응 플랫폼이다.

```text
[발굴] Soli AI 정기 안부 감지 / 담당자 등록 / 본인 요청 대화
  -> [판단 보조] Soli AI 위기 신호 요약 + 응답 공백·위험 표현 근거 + 모바일 신분증 확인
  -> [집행] 담당자 대시보드 우선순위 확인 + 지원 승인
  -> [기록] OmniOne Chain에 지원 확인 VC 메타데이터·지원 완료 증명 해시 기록 + txId 표시
```

### 핵심 기능

| 기능 | 대상 | 설명 |
|:---|:---|:---|
| Soli AI 정기 안부 | 독거노인·취약계층·담당자 관리 대상자 | 대상자 동의 후 Soli가 매일/격일로 안부를 묻고, 감정 표현·위기 키워드·응답 공백을 기반으로 위기 신호를 요약한다. 최종 판단은 담당자가 한다. |
| 담당자 대시보드 | 복지 담당자 | 오늘 우선 확인이 필요한 사람을 위기 신호 수준과 AI 근거 요약을 바탕으로 확인한다. |
| OmniOne 신뢰 확인 | 대상자·수급자 | 가입·로그인과 집행 직전 본인 확인을 통해 서비스 진입과 지원 집행의 신뢰를 확보한다. |
| 온체인 기록 | 지자체·기업 | VC 발급·판단·집행의 증명 해시를 기록해 사후 검증 가능성을 높인다. |
| Soli 복지 매칭 | 위기 대상자 | 감지 이후 지원 연결을 돕는 보조 기능. 기본 조건과 선택형 상황 체크리스트를 바탕으로 받을 가능성이 있는 복지를 안내한다. |

### 초기 유입·감지 경로

| 경로 | 첫 신호 | 운영 방식 |
|:---|:---|:---|
| 담당자 등록 | 기존 관리 대상자의 정기 안부 확인 필요 | 대상자 본인 동의 후 Soli 정기 안부 대화를 연결하고 대시보드에서 추적한다. |
| Soli 정기 안부 감지 | 응답 공백·위기 키워드·감정 악화 | Soli가 매일/격일 안부 대화에서 위기 신호를 감지하고 담당자 우선 확인 목록에 반영한다. |
| 본인 요청 대화 | 사용자가 직접 어려움이나 긴급 도움을 표현 | Soli가 위기 신호를 요약하고, 필요한 경우 담당자 확인 또는 지원 연결로 이어진다. |
| 지원 연결 보조 | 감지 이후 복지 정보 확인 필요 | Soli가 기본 조건과 상황 체크리스트를 함께 받아 받을 가능성이 있는 복지를 안내한다. |

### 모바일 신분증 활용 지점

OmniOne CX 표준 인증창 연동으로 검증 결과 수신 (해커톤 제공 SDK).

| 사용 시점 | 역할 | 효과 |
|:---|:---|:---|
| 서비스 최초 가입·로그인 | 사용자 신원 확인 | 안전한 서비스 진입, 신뢰 기반 구축 |
| 지원 집행 직전 | 수급자 본인 확인 | 부정 수급 방지 |

### OmniOne Chain 활용 구조

no-one-alone은 복지 데이터 원문을 블록체인에 올리지 않는다. 대신 VC 발급·승인·집행 이벤트의 해시만 OmniOne Chain에 기록해, 지자체·후원 기업·감사 주체가 같은 기록을 기준으로 사후 검증할 수 있게 한다.

| 이벤트 | 온체인 기록 | 검증 가치 |
|:---|:---|:---|
| 지원 확인 VC 발급 | VC 메타데이터 해시 + 발급 타임스탬프 | 지원이 실제 대상자에게 연결되었음을 사후 검증 |
| 담당자 승인 | 승인자 해시 + 케이스 해시 + 승인 시각 | 누가 어떤 근거로 승인했는지 사후 검증 |
| 지원 집행 | 수급자 DID 해시 + 지원 종류·금액 해시 + txId | 지자체·보험사·감사 주체가 동일 txId로 집행 사실을 각자 독립 확인 — 다기관 사후 증빙 비용 절감 |

**개인정보 경계**: MVP 데모에서는 실제 주소·상세 상황 설명을 저장하지 않고, 지역 코드·해시·데모 데이터를 중심으로 시연한다. OmniOne Chain에는 DID 해시·이벤트 해시·지원 완료 증명 해시만 기록한다.

### 판단 책임 원칙

- AI는 수급 여부나 지원 여부를 자동 결정하지 않는다.
- Soli는 위기 키워드, 응답 공백, 감정 변화 등 담당자가 확인할 근거를 요약한다. 복지 매칭 결과는 지원 연결이 필요할 때 보조 정보로 표시한다.
- 담당자 대시보드는 "왜 고위험인지"를 함께 표시하고, 최종 승인과 집행은 사람이 수행한다.
- MVP 개발 단계에서는 위험 스코어보다 설명 가능한 근거 표시를 우선 구현한다.

### 개발·시연 범위

현재 구현된 웹앱, API, DB, OmniOne 연동은 없다. 5월 제안서 제출 후 개발·시연 대상으로 선정되면, 2026-07-01 ~ 2026-09-21 기간 동안 아래 범위를 MVP로 구현한다.

| 범위 | 구현 목표 | 시연 장면 |
|:---|:---|:---|
| Soli 챗봇 | 위기 키워드 감지와 위기도 스코어 출력 | 사용자가 위험 문장을 입력하면 스코어가 상승 |
| Soli 복지 매칭 | 지원 연결 보조 카드 | 감지 이후 필요한 경우 받을 가능성이 있는 복지 표시 후 복지로 외부 링크 연결 |
| 담당자 대시보드 | 위기 우선순위 리스트와 상세 요약 | 담당자가 고위험 케이스를 확인 |
| OmniOne CX | 해커톤 제공 SDK·가이드 기반 표준 인증창 연동 | 가입·로그인, 집행 직전 수급자 확인 |
| 지원 승인 | 지원 종류 선택과 승인 상태 변경 | 담당자가 지원 집행을 승인 |
| OmniOne Chain | 지원 확인 VC 메타데이터·지원 완료 증명 해시 기록 + txId 표시 | 지원 완료 증명의 txId 확인 |

**MVP 제외 범위**: 개인 자격 확인(공공 마이데이터 이용기관 승인 필요 — 파일럿 이후), 실제 행정 시스템 연동, 보험사 대상 비식별 데이터 상품화·공동 실증 계약, 실제 복지금 지급·정산, 실사용자 개인정보 수집.

**내부 우선순위**: 가점 +10% 확보가 목표이므로 Open DID VC 발급·VP 검증과 OmniOne Chain txId 표시는 MVP 필수 성공 기준이다. 필수 데모는 Soli 감지 → 담당자 승인 → 집행 직전 본인 확인 → 지원 확인 VC 발급 → VP 검증 → 지원 완료 증명 txId 표시까지 연결한다. 외부 SDK·네트워크 연동 지연에 대비해 Mock 인증·데모 데이터·녹화 영상을 준비하되, 제출 목표는 최소 1개 실제 VC/VP/txId 성공이다.

## 5. 제안내용의 차별성

**One-line message**: 기존 시스템이 놓치는 위기 신호를 Soli 정기 안부와 AI 근거 요약으로 더 빨리 연결한다.

### 기존 시스템과의 차이

| 구분 | 기존 복지 시스템 | 고독사 보험 | no-one-alone |
|:---|:---|:---|:---|
| 발굴 방식 | 행정 데이터 중심, 정기 수집 | 보험 가입 이후 사후 보장 중심 | Soli AI 상시 감지 + AI 근거 요약 |
| 사각지대 | 행정 데이터 밖 현장 신호 반영이 어려움 | 가입 전 위험 신호 파악이 어려움 | 24/7 Soli 대화로 신호 포착 |
| 신청 방식 | 본인 신청 필수 | 해당 없음 | Soli 감지·담당자 등록으로 먼저 위기 신호를 연결하고, 복지 매칭은 지원 연결을 보조 |
| 집행 투명성 | 기관 내부 기록 중심 | 해당 없음 | 지원 완료 증명 해시로 사후 검증 보완 |
| 담당자 업무 | 다수 대상 수동 확인 부담 | 해당 없음 | 위기도 우선순위 기반 1차 분류 |
| 데이터 축적 | 기관별로 분절 | 보험 계약 데이터 중심 | 대화 신호-응답 공백-담당자 처리 결과를 연결한 사회적 고립 신호 누적 |

### 기술 차별성

- OmniOne CX는 해커톤 제공 SDK·가이드를 기준으로 표준 인증창 연동부터 구현한다. 연동 지연에 대비해 인증 완료 Mock과 실제 SDK 연동 모드를 분리한다.
- Open DID는 지원 확인 VC 발급·VP 검증에 활용한다. MVP 개발 단계에서는 VC 발급 데모 + VP 검증 흐름으로 범위를 제한한다.
- OmniOne Chain은 개인정보 원문이 아니라 지원 확인 VC 메타데이터·지원 완료 증명 해시·txId만 기록한다. DB 로그를 대체하는 것이 아니라 지자체·보험회사·CSR 후원자·감사 주체가 함께 신뢰할 수 있는 사후 증명 레이어다.
- Soli AI는 단순 상담 챗봇이 아니라 복지 담당자에게 전달할 위기 신호를 정리하는 첫 접점이다.
- 개인 자격 확인과 공공 마이데이터 연계는 MVP 범위에서 제외한다. MVP에서는 한국사회보장정보원 복지서비스 API로 받을 가능성이 있는 복지를 매칭하고, 실제 개인 자격 확인은 파일럿 이후 지자체 또는 승인된 이용기관 협력 단계에서 추진한다.

### 신뢰 문제별 기술 적용

| 심사위원 질문 | 적용 기술 | 답변 |
|:---|:---|:---|
| AI 오판으로 낙인이 찍힐 수 있지 않나? | Soli 근거 요약 + 담당자 최종 판단 | AI는 판정자가 아니라 위기 신호 요약 도구다. 스코어 근거(어떤 키워드, 며칠 응답 공백)를 함께 표시하고, 지원 여부는 반드시 담당자가 최종 승인한다. |
| 서류 없이 복지 자격을 어떻게 확인하나? | 단계 분리 | MVP에서는 개인 자격을 확정하지 않고, 한국사회보장정보원 API로 받을 가능성이 있는 복지를 매칭한 뒤 복지로 외부 신청으로 연결한다. 개인 자격 확인은 파일럿 이후 지자체 또는 승인된 이용기관 협력 단계에서 추진한다. |
| 블록체인이 꼭 필요한가? | OmniOne Chain 해시 기록 | 개인정보 원문이 아니라 지원 완료 증명 해시만 기록한다. 지자체 담당자·보험사 심사 담당자·감사 주체가 동일한 txId를 기준으로 "이 수급자가 이 날 이 지원을 실제로 받았다"는 사실을 각자 독립적으로 확인할 수 있어, 다기관 사후 증빙 비용을 낮춘다. |
| 보험회사에 개인 정보가 넘어가나? | 집계·비식별 데이터 원칙 | 개인 단위 판매는 금지하고, 지역·연령대·위험 패턴 단위 통계와 공동 실증 결과만 제공한다. |

**핵심 해자 (Moat)**: Soli 대화 패턴, 응답 공백 이력, 담당자 처리 결과가 함께 쌓일수록 기존 공공 데이터로는 포착하기 어려운 사회적 고립 신호 데이터셋이 축적된다. 이 데이터는 지자체 선제 대응 정책과 비식별 통계 기반 공동 실증의 기반이 되어 경쟁자 진입 장벽이 된다.

- 온체인 증명으로 복지 집행과 CSR 기부의 투명성을 강화한다.

### 사업화 및 확장 가능성

**One-line message**: 지자체에는 발굴·우선순위화 도구를 팔고, 보험회사는 비식별 통계 기반 공동 실증 파트너로 확장한다.

### 핵심 고객 1: 지자체 B2G SaaS

행정복지센터 담당자용 AI 우선순위화 도구를 기초지자체 단위 연간 라이선스로 공급한다.

| 단계 | 규모 | 연 매출 |
|:---|:---|:---|
| 파일럿 | 1개 기초지자체 | 무상 PoC 3개월 → 성과 후 협약 연 2,000만원 |
| 3년 목표 | 10개 기초지자체 | 연 2,500만원/기관 → 연 2.5억 |
| 전국 TAM | 226개 기초지자체 | 연 2,500만원/기관 → 연 56억 (시장 규모 기준) |

진입 전략: 해커톤 입상 후 고독사 예방·복지 사각지대 발굴 의지가 있는 1개 기초지자체와 PoC를 추진한다. 초기에는 무상 또는 저비용 시범사업으로 위기 신호 확인 시간, 담당자 만족도, 지원 연결률을 측정하고, 성과지표 확보 후 협약·조달 채널을 검토한다.

### 확장 파트너: 보험회사 공동 실증

보험회사에는 개인 단위 위험 판정 데이터가 아니라, 고독사 예방형 사회안전망 상품을 고도화하기 위한 집계·비식별 위험 신호 통계와 공동 실증 결과를 제공한다.

| 단계 | 제공 가치 | 수익화 방식 |
|:---|:---|:---|
| PoC | 지역·연령대별 고립 신호, 응답 공백 패턴 검증 | PoC 용역비 또는 공동 실증 |
| 상품 고도화 | 고독사 예방형 사회안전망 상품 설계 참고 지표 제공 | 공동 연구 또는 리포트 구독 |
| 운영 고도화 | 가입자 동의 기반 안부 확인, 위험 신호 알림, 지자체 연계 | 보험사 솔루션 이용료 |

**보험사가 필요한 이유**: 기존 고독사 보험은 예방형 서비스와 지역 단위 위험 신호 데이터가 부족해 상품 확산이 제한적이다. no-one-alone이 축적하는 비식별 고립 신호와 응답 공백 패턴은 보험사가 개인을 선별하는 용도가 아니라 예방형 상품과 공공 연계 모델을 설계하는 참고 지표로 활용된다.

- Soli 정기 안부·담당자 등록·지원 연결 보조 유입 경로 표
- 모바일 신분증 활용 2지점 표 + OmniOne CX 표준 인증창 연동 설명
- 한국사회보장정보원 복지서비스 API 기반 매칭 화면
- 신뢰 문제별 기술 적용 표
- 기존 복지 시스템 대비 no-one-alone 비교표
- 개발·시연 추진 일정표
- 개인정보·온체인 데이터 처리 원칙

## 8. 기타

### 팀 역량

**One-line message**: AI 기반 위기 분류와 DID·온체인 증명 흐름을 빠르게 구현·검증하는 팀.

| 역할 | 담당 | 강점 |
|:---|:---|:---|
| 팀장 / 풀스택 개발 | 개발 총괄 | Next.js, NestJS 기반 웹 서비스 구현, API 설계, 블록체인 연동 구조 학습 및 시연 설계 |
| AI·백엔드 개발 | 공동 개발 | Soli 챗봇, 위기 스코어링, API 구현 |

**팀 역량 증거**:
- Next.js, NestJS 기반 웹 서비스 구현 역량
- Soli 챗봇, 담당자 대시보드, 증명 해시 표시까지 이어지는 단일 시연 플로우 설계 완료

### 주요 리스크와 대응

| 리스크 | 대응 |
|:---|:---|
| OmniOne CX SDK 연동 문제 | 해커톤 제공 표준 인증창 연동 방식으로 범위를 제한. 미지원 시 인증 완료 Mock으로 fallback |
| AI 응답 품질 불안정 | 룰 기반 위기 키워드 스코어링을 fallback으로 구현 |
| AI 오판·낙인 우려 | AI는 최종 판정자가 아니라 근거 요약 도구로 제한하고, 담당자 최종 승인 구조를 명시 |
| 개인 자격 확인·민감정보 우려 | MVP에서는 개인 행정정보를 조회하지 않는다. 한국사회보장정보원 복지서비스 API로 복지 후보만 매칭하고, 개인 자격 확인은 파일럿 이후 승인 기관 협력으로 추진한다. |
| 개인정보 우려 | 실사용자 데이터 없이 데모 데이터와 DID 해시만 사용. 보험회사에는 집계·비식별 데이터만 제공 |
| 범위 과대 | 핵심 시연을 Soli 감지, 담당자 승인, 온체인 기록으로 집중 |
| 발표 당일 네트워크 장애 | 로컬 데모 데이터와 녹화 영상 준비 |
