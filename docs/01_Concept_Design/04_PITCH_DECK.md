### 시스템이 실패하는 3가지 구조적 메커니즘

현장에서 반복되는 비극은 제도의 부재가 아니라 시스템의 구조적 맹점 때문이다.

**1. 신청주의 장벽**: 복지는 본인이 직접 신청해야 수급이 가능하다. 그러나 정신질환이나 발달장애가 있는 대상자에게 진료 기록, 진단서, 소득 증빙을 직접 준비하는 일은 사실상 불가능에 가까운 벽이다. "신청이라는, 누군가에게는 너무나 쉽고 당연한 권리가 누군가에게는 넘을 수 없는 벽이기도 합니다." (그것이알고싶다 나레이션)

**2. 수급자 배제 역설 (Proxy Trap)**: 서울 방배동에서 발달장애인 아들이 수개월간 어머니의 사망 사실을 혼자 안고 거리에서 노숙한 사건에서, 이 가구는 주거 급여를 수급 중이라는 이유로 보건복지부 위기가구 발굴 대상에서 제외되어 있었다. 이미 도움을 받고 있다고 간주된 순간, 더 심각한 위기가 감지 시스템 밖으로 빠져나간다.

**3. 아동복지 단절 절벽**: 시설에서 보호받던 아이들은 만 18세가 되면 퇴소해 원래의 취약한 가정으로 돌아갈 수밖에 없는 구조적 단절이 존재한다. 경남 창원에서 발견된 정신질환 모녀 사건에서 딸은 시설을 퇴소한 후 다시 어머니에게로 돌아갔고, 두 사람 모두 통장 잔고 7,000원의 상황에서도 공적 지원을 받지 못했다.

## 4. 제안내용의 상세 설명

**One-line message**: 사회복지사의 눈이 닿지 않는 곳을, AI가 찾고, 복지가 닿고, 블록체인이 증명한다.

### 서비스 정의

no-one-alone은 **발굴 → 판단 → 집행 → 기록**을 하나로 연결하는 복지 위기 대응 플랫폼이다.

```text
발굴   현장 QR 자가 가입 · 담당자 등록 · Soli AI 정기 안부 감지 · 본인 요청 대화
  →
판단   Soli AI 위기 신호 요약 + 응답 공백·위험 표현 근거 → 담당자 최종 판단
  →
집행   OmniOne CX 집행 전 본인 재확인 → 담당자 지원 승인
  →
기록   OmniOne Chain — VC 메타데이터 해시 + 집행 이벤트 해시 + txId 표시
```

### Soli AI의 역할 — 사회복지사의 눈과 귀

Soli는 단순한 상담 챗봇이 아니다. 복지대상자와 응급 돌봄 수요가 빠르게 늘어나는 상황에서 담당자가 오늘 당장 문을 두드려야 할 한 사람을 찾을 수 있도록 돕는 라스트 마일(Last Mile) 지원 도구다.

| 역할 | 설명 |
|:---|:---|
| 상시 감지 | 격월 행정 데이터 수집 대신 매일 안부 대화로 위기 신호를 실시간 포착 |
| 정성 신호 포착 | 전기료 체납 같은 숫자 데이터가 놓치는 감정 변화·대화 단절·응답 공백을 감지 |
| Proxy Trap 차단 | 이미 수급 중인 대상자도 Soli 대화 패턴에서 위기 신호가 감지되면 담당자 우선 확인 목록에 반영 |
| 판단 보조 | 어떤 맥락에서 신호가 감지되었는지 요약 → 담당자 빠른 의사결정 지원. AI는 판단을 대체하지 않음 |

### 핵심 기능

| 기능 | 대상 | 설명 |
|:---|:---|:---|
| Soli AI 정기 안부 | 독거노인·취약계층·담당자 관리 대상자 | 대상자 동의 후 Soli가 매일/격일로 안부를 묻고, 감정 표현·위기 키워드·응답 공백을 기반으로 위기 신호를 요약한다. 최종 판단은 담당자가 한다. |
| 담당자 대시보드 | 복지 담당자 | Soli가 담당자 관리 대상의 안부 신호를 스캔해, 담당자가 출근하면 "오늘 당장 방문해야 할 상위 5명"을 위기 신호 수준·AI 근거 요약과 함께 즉시 확인할 수 있다. AI가 대체하는 것이 아니라, 담당자가 매일 가장 중요한 한 사람에게 집중할 수 있도록 효율적으로 보조한다. |
| OmniOne 신뢰 확인 | 대상자·수급자 | 가입·로그인과 집행 직전 본인 확인을 통해 서비스 진입과 지원 집행의 신뢰를 확보한다. |
| 온체인 기록 | 지자체·기업 | VC 발급·판단·집행의 증명 해시를 기록해 사후 검증 가능성을 높인다. |
| Soli 복지 매칭 | 위기 대상자 | 감지 이후 지원 연결을 돕는 보조 기능. 기본 조건과 선택형 상황 체크리스트를 바탕으로 받을 가능성이 있는 복지를 안내한다. |

| 경로 | 첫 신호 | 운영 방식 |

| Soli 정기 안부 감지 | 응답 공백·위기 키워드·감정 악화 | Soli가 매일/격일 안부 대화에서 위기 신호를 감지하고 담당자 우선 확인 목록에 반영한다. |

| 지원 연결 보조 | 감지 이후 복지 정보 확인 필요 | Soli가 기본 조건과 상황 체크리스트를 함께 받아 받을 가능성이 있는 복지를 안내한다. |

### 모바일 신분증 활용 지점

OmniOne CX 표준 인증창 연동으로 검증 결과 수신 (해커톤 제공 SDK).

| 사용 시점 | 역할 | 효과 |
|:---|:---|:---|
| 서비스 최초 가입·로그인 | 사용자 신원 확인 | 안전한 서비스 진입, 신뢰 기반 구축 |
| 지원 집행 직전 | 수급자 본인 확인 | 부정 수급 방지 |

### OmniOne Chain 활용 구조

no-one-alone은 복지 데이터 원문을 블록체인에 올리지 않는다. 대신 VC 발급·승인·집행 이벤트의 해시만 OmniOne Chain에 기록해, 지자체·후원 기업·감사 주체가 같은 기록을 기준으로 사후 검증할 수 있게 한다.

| 이벤트 | 온체인 기록 | 검증 가치 |
|:---|:---|:---|
| 지원 확인 VC 발급 | VC 메타데이터 해시 + 발급 타임스탬프 | 지원이 실제 대상자에게 연결되었음을 사후 검증 |
| 담당자 승인 | 승인자 해시 + 케이스 해시 + 승인 시각 | 누가 어떤 근거로 승인했는지 사후 검증 |
| 긴급 확인 필요 신호 | 응답 공백 시간 + 위기 키워드 빈도 해시 + 감지 타임스탬프 | Soli 응답 단절 72시간 이상·위기 키워드 복합 감지 시 담당자의 현장 확인 판단을 돕는 보조 기록으로 활용 |
| 지원 집행 | 수급자 DID 해시 + 지원 종류·금액 해시 + txId | 지자체·보험사·감사 주체가 동일 txId로 집행 사실을 각자 독립 확인 — 다기관 사후 증빙 비용 절감 |

**개인정보 경계**: MVP 데모에서는 실제 주소·상세 상황 설명을 저장하지 않고, 지역 코드·해시·데모 데이터를 중심으로 시연한다. OmniOne Chain에는 DID 해시·이벤트 해시·지원 완료 증명 해시만 기록한다.

## 5. 제안내용의 차별성

**One-line message**: 신청을 기다리지 않고 먼저 문을 두드린다 — AI 감지, 블록체인 증명, 담당자 연결을 하나의 파이프라인으로.

### 기존 시스템과의 차이

| 구분 | 기존 복지 시스템 | no-one-alone |
|:---|:---|:---|
| 발굴 방식 | 행정 데이터 중심, 정기 수집 | Soli AI 상시 감지 + AI 근거 요약 |
| 사각지대 | 행정 데이터 밖 현장 신호 반영이 어려움 | 24/7 Soli 대화로 신호 포착 |
| 신청 방식 | 본인 신청 필수 | Soli 감지·담당자 등록으로 먼저 위기 신호를 연결하고, 복지 매칭은 지원 연결을 보조 |
| 집행 투명성 | 기관 내부 기록 중심 | 지원 완료 증명 해시로 사후 검증 보완 |
| 담당자 업무 | 다수 대상 수동 확인 부담 | 위기도 우선순위 기반 1차 분류 |
| 데이터 축적 | 기관별로 분절 | 대화 신호-응답 공백-담당자 처리 결과를 연결한 사회적 고립 신호 누적 |

### 기술 차별성

- OmniOne CX는 해커톤 제공 SDK·가이드를 기준으로 표준 인증창 연동부터 구현한다. 연동 지연에 대비해 인증 완료 Mock과 실제 SDK 연동 모드를 분리한다.
- Open DID는 지원 확인 VC 발급·VP 검증에 활용한다. MVP 개발 단계에서는 VC 발급 데모 + VP 검증 흐름으로 범위를 제한한다.
- OmniOne Chain은 개인정보 원문이 아니라 지원 확인 VC 메타데이터·지원 완료 증명 해시·txId만 기록한다. DB 로그를 대체하는 것이 아니라 지자체·보험회사·감사 주체가 함께 신뢰할 수 있는 사후 증명 레이어다.
- Soli AI는 단순 상담 챗봇이 아니라 복지대상자와 응급 돌봄 수요가 늘어나는 현장에서 담당자의 눈과 귀로, 대화 맥락과 응답 공백에서 위기 신호를 먼저 걸러내는 첫 접점이다. "단순히 위험함이라고 알려주는 것이 아니라, 어떤 대화 맥락에서 신호가 감지되었는지 요약해 담당자의 빠른 의사결정을 돕는다."
- 개인 자격 확인과 공공 마이데이터 연계는 MVP 범위에서 제외한다. MVP에서는 한국사회보장정보원 복지서비스 API로 받을 가능성이 있는 복지를 매칭하고, 실제 개인 자격 확인은 파일럿 이후 지자체 또는 승인된 이용기관 협력 단계에서 추진한다.

### 신뢰 문제별 기술 적용

| 심사위원 질문 | 적용 기술 | 답변 |
|:---|:---|:---|
| AI 오판으로 낙인이 찍힐 수 있지 않나? | Soli 근거 요약 + 담당자 최종 판단 | AI는 판정자가 아니라 위기 신호 요약 도구다. 스코어 근거(어떤 키워드, 며칠 응답 공백)를 함께 표시하고, 지원 여부는 반드시 담당자가 최종 승인한다. |
| 서류 없이 복지 자격을 어떻게 확인하나? | 단계 분리 | MVP에서는 개인 자격을 확정하지 않고, 한국사회보장정보원 API로 받을 가능성이 있는 복지를 매칭한 뒤 복지로 외부 신청으로 연결한다. 개인 자격 확인은 파일럿 이후 지자체 또는 승인된 이용기관 협력 단계에서 추진한다. |
| 블록체인이 꼭 필요한가? | OmniOne Chain 해시 기록 | txId 하나로 지자체(행정), 보험사(보상 심사), 후원 기업(투명성 보고)이 각자의 시스템에서 즉시 독립 검증할 수 있다. 기존에는 세 기관이 서로 공문과 서류를 주고받아야 했던 다기관 사후 증빙 비용을 기록 시점부터 구조적으로 절감한다. |
| 방문해도 문을 안 열면 어떻게 하나? | Soli 사전 완충 + 긴급 확인 필요 신호 기록 + 접근 브리핑 | 세 단계로 대응한다. (1) 담당자 방문 전 Soli가 먼저 대화를 시도해 "구청 선생님이 걱정돼서 오셨다"는 정서적 완충을 만든다. (2) Soli 응답 공백 72시간 이상 + 위기 키워드 복합 감지 시 이를 긴급 확인 필요 신호로 남겨, 담당자가 기존 행정·법 체계 안에서 현장 확인 여부를 판단할 때 참고할 수 있게 한다. (3) 대시보드가 과거 대화 패턴을 요약해 "이 대상자는 '복지' 단어에 거부감, '건강 체크' 명목 시 응답률 상승" 같은 맞춤형 접근 브리핑을 제공한다. NOA는 강제 개입을 결정하지 않고, 담당자의 판단과 현장 대응을 보조한다. |
| 보험회사에 개인 정보가 넘어가나? | 집계·비식별 데이터 원칙 | 보험사에 제공되는 것은 특정 개인의 위험도가 아니라 권역·연령대·가구 유형 단위로 충분한 표본 수를 충족한 집계 통계다. k-익명성 등 재식별 방지 기준을 만족하지 못하는 작은 집단은 제공하지 않는다. 개인을 선별하는 도구가 아니라 고독사 보험의 보험료 산정, 위험 지역 분석, 예방형 부가서비스 설계를 위한 참고 지표로만 활용한다. |

**데이터 경쟁 우위**: Soli 대화 패턴, 응답 공백 이력, 담당자 처리 결과가 함께 쌓일수록 기존 공공 데이터로는 포착하기 어려운 사회적 고립 신호 데이터셋이 축적된다. 이 데이터는 지자체 선제 대응 정책과 비식별 통계 기반 공동 실증의 기반이 되어 경쟁자 진입 장벽이 된다.

## 6. 기대효과

**One-line message**: 사각지대 발견부터 지원 증명까지 — 감지 속도는 빨라지고, 집행 신뢰는 온체인으로 남는다.

### 기대효과 요약

| 효과 | 변화 |
|:---|:---|
| 사회적 효과 | 고독사와 고립 위기를 사고 이후가 아니라 신호 단계에서 포착해 선제 개입 가능성을 높인다. |
| 행정 효과 | 담당자가 수많은 대상자 중 오늘 우선 확인해야 할 사람을 먼저 볼 수 있어 현장 확인의 우선순위가 명확해진다. |
| 신뢰 효과 | 지원 집행 기록을 해시와 txId로 남겨 지자체, 후원 기업, 감사 주체가 같은 기록을 기준으로 사후 검증할 수 있다. |
| 시장 효과 | 축적된 비식별 고립 신호는 고독사 보험의 위험률 분석, 보험료 산정, 예방형 부가서비스 설계에 활용될 수 있다. |
| 관계 회복 | 고립된 대상자에게 Soli의 정기 안부는 "누군가 나를 찾고 있다"는 신호가 된다. |

### 단계별 측정 지표

| 단계 | 목표 | 측정 방법 |
|:---|:---|:---|
| 개발·시연 단계 | OmniOne CX 가입·로그인 → Soli AI 감지 → 담당자 승인 → 집행 직전 본인 확인 → VC/txId 기록 시연 완성 | E2E 시나리오 성공 여부, 위기 신호 근거와 txId 표시 여부 |
| 2027 파일럿 | 위기 신호 확인 시간 단축, 담당자 업무 효율 검증 | 위기 신호 발생부터 담당자 확인까지의 시간, 담당자 설문 |
| 3년 임팩트 | 발굴 후 지원 연결률 개선, 상시 위기 감지 체계 검증 | 파일럿 지자체 집행 기록, Soli 대화·응답 공백 감지 빈도 |

### 파일럿 검증 가설 (3년 임팩트 지표)

| 지표 | 현재 기준 | 3년 가설 | 측정 방법 |
|:---|:---:|:---:|:---|
| 발굴 후 지원 연결률 | 약 43.6% | 70% 이상 | 파일럿 지자체 담당자 집행 기록 |
| 위기정보 수집 주기 | 격월 | 상시 | Soli 대화 및 응답 공백 감지 빈도 |
| 1차 확인·분류 시간 | 수주 | 24시간 이내 | 위기 신호 발생 → 담당자 1차 확인·분류 시간 |
| 초기 확인 업무 부담 | 건별 수동 확인 | 50% 감소 | 케이스당 확인 소요 시간, 담당자 설문 |

*위 수치는 3년 내 달성을 목표로 하는 파일럿 검증용 가설 KPI이며, 해커톤 개발·시연 단계에서 보장하는 수치가 아니다. MVP 단계에서는 이 KPI를 달성했다고 주장하지 않고, Soli 감지 로그, 담당자 확인 이력, VC/txId 기록, 담당자 피드백처럼 이후 파일럿에서 측정 가능한 구조를 만드는 데 집중한다.*

> "신청할 수 없는 사람들을 위해, 국가가 먼저 문을 두드리는 시스템."

> "AI가 신호를 찾고, 복지가 손을 잡으며, 블록체인이 투명성을 증명합니다."

### 담당자 Pain Point 해결 — B2G 현장 수용성 근거

지자체가 시스템을 도입해도 현장 담당자가 쓰지 않으면 의미가 없다. 아래 세 가지 Pain Point 해결이 실질적인 수용성 근거다.

| Pain Point | 현실 | NOA 대응 |
|:---|:---|:---|
| "내가 놓쳐서 사고 나면?" — 심리적 중압감 | 사고 발생 시 관리 소홀 화살이 담당자 개인에게 쏟아짐 | Soli 상시 모니터링 기록이 "할 수 있는 최선을 다했음"을 데이터로 증명. AI 위기 스코어가 담당자 개인 판단이 아닌 시스템과의 공동 판단 체계로 심리적 책임을 분산 |
| "서류 만들다 하루가 다 가요" — 행정 늪 | 기록·증빙·보고서 작성 부담이 큼 | MVP에서는 Soli 대화 요약과 방문 메모 초안을 제공하고, 파일럿 이후 상담 일지·보고서 초안 자동화로 확장한다. OmniOne Chain txId는 감사·보고 시 집행 무결성 확인 근거로 활용 |
| "매번 처음부터 물어야 해요" — 히스토리 단절 | 담당자 교체·휴가 시 대상자 특성·히스토리 유실 | Soli가 대상자 습관·선호·최근 감정 상태를 축적 → 신규 담당자 방문 전 "이분은 최근 예민한 상태, 먼저 위로 권장" 같은 맞춤 브리핑 제공 (디지털 인계장) |

핵심 메시지: "블록체인과 AI가 단순 감시 도구가 아니라, 지긋지긋한 사후 증빙 서류 작업을 줄이고 돌봄에 집중할 수 있게 해주는 도구"

### 사업화 및 확장 가능성

**One-line message**: 지자체에는 발굴·우선순위화 도구를 팔고, 보험회사에는 고독사 보험의 리스크 가격 산정에 필요한 비식별 통계 리포트를 제공한다.

### 핵심 고객 1: 지자체 B2G SaaS

행정복지센터 담당자용 AI 우선순위화 도구를 기초지자체 단위 연간 라이선스로 공급한다.

| 단계 | 규모 | 연 매출 |
|:---|:---|:---|
| 파일럿 | 1개 기초지자체 | 무상 PoC 3개월 → 성과 후 협약 연 2,000만원 |
| 3년 목표 | 10개 기초지자체 | 연 2,500만원/기관 → 연 2.5억 |
| 전국 TAM | 226개 기초지자체 | 연 2,500만원/기관 → 연 56억 (시장 규모 기준) |

진입 전략: 해커톤 입상 후 고독사 예방·복지 사각지대 발굴 의지가 있는 1개 기초지자체와 PoC를 추진한다. 초기에는 무상 또는 저비용 시범사업으로 위기 신호 확인 시간, 담당자 만족도, 지원 연결률을 측정하고, 성과지표 확보 후 협약·조달 채널을 검토한다.

### 확장 파트너: 보험회사 리스크 데이터 PoC

고독사 보험은 임대주택 내 고독사 발생 시 임대인이 부담하는 특수청소, 유품 정리, 시신 부패로 인한 원상복구 비용, 임대료 손실 등을 보상하는 상품이다. 일본에서는 고령화와 함께 일반화된 보험 영역으로 성장했지만, 국내는 2022년 이후 일부 손해보험사가 상품을 출시한 초기 시장이다.

국내 보험사가 가장 필요로 하는 것은 특정 개인의 위험도 데이터가 아니라, 보험료 산정과 지역별 위험률 분석에 활용할 수 있는 집계·비식별 리스크 데이터다. no-one-alone은 Soli 응답 공백, 고립 신호, 담당자 처리 결과를 지역·연령대·가구 유형 단위로 집계해 고독사 보험의 가격 산정, 위험 지역 분석, 예방형 서비스 설계에 활용할 수 있는 데이터 기반을 제공한다.

| 단계 | 제공 가치 | 수익화 방식 |
|:---|:---|:---|
| PoC | 권역·연령대·가구 유형별 고립 신호, 응답 공백 패턴 검증 | PoC 용역비 또는 공동 실증 |
| 리스크 가격 산정 | 고독사 보험 보험료 산정·위험 지역 분석 참고 지표 제공 | 비식별 리스크 리포트 구독 또는 공동 연구 |
| 예방형 부가 서비스 | 가입자 동의 기반 안부 확인, 위험 신호 알림, 지자체 연계 | 보험사 솔루션 이용료 |

**보험사가 필요한 이유**: 고독사 보험은 사고 빈도와 원상복구 비용을 예측해야 보험료를 정할 수 있지만, 국내 시장은 아직 표본과 지역별 위험률 데이터가 부족하다. no-one-alone이 축적하는 비식별 고립 신호와 응답 공백 패턴은 보험사가 개인을 선별하는 용도가 아니라, 권역 단위 위험률과 예방형 부가서비스를 설계하는 참고 지표로 활용된다.

### 이후 확장 경로

| 레이어 | 수익 모델 | 진입 조건 |
|:---|:---|:---|
| Phase 2 가족 안부 케어 | 가족 초대 기반 유료 안부 케어 또는 지자체 돌봄 패키지 | 취약계층 발굴 MVP과 파일럿 성과 검증 후. 대상자 본인 동의 기반으로만 운영 |
| 데이터 API | 익명 집계 리스크 데이터 구독 | 2029+ 장기 검토. 수천 명 이상 데이터 축적 후 k-익명성 등 재식별 방지 기준을 충족하는 범위에서만 고독사 보험·사회안전망 상품용 집계 통계로 제공 |

**개인정보 원칙**: 보험·비식별 데이터 상품 수익 모두 집계·비식별 처리된 데이터만 활용한다. 개인 단위 데이터 판매는 없으며, 표본 수가 작은 지역·연령·가구 유형 조합은 재식별 위험이 있으므로 제공하지 않는다.

### 성장 로드맵

| 단계 | 기간 | 목표 |
|:---|:---|:---|
| 생존 | 2026 | 해커톤 입상 + MVP 개발·시연 |
| 검증 | 2027 | 1개 지자체 파일럿 + 3~5개 지자체 확산 협의 + 보험사 1개 공동 실증 |
| 성장 | 2028 | 지자체 유료 전환 + 보험사 정식 계약 |
| 확장 | 2029+ | Phase 2 가족 안부 케어, 국내 복지·보험 생태계 전반 확장 |
