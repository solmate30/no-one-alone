**현재 상태**: 제안서 작성 단계 (구현 없음)
- 2026-05-31 온라인 모집 마감 전 공모 제안서 제출이 1차 목표
- 현재 문서는 제안서 본문, 발표 흐름, MVP 구현 계획을 정리하기 위한 설계 자료
- Next.js + TypeScript 기반 단일 반응형 웹 MVP와 담당자 대시보드는 MVP 구현 예정

## 3. 핵심 기능 명세

Solmate 사용자는 서비스 최초 가입·로그인 시 OmniOne CX 본인확인을 완료한다. 핵심 기능은 Soli 정기 안부와 대화 신호를 통해 위험한 취약계층을 선제 감지하고, 담당자가 오늘 우선 돌봐야 할 대상을 확인하게 하는 것이다. 신청주의 장벽을 넘을 수 없는 대상자를 위해 Soli가 먼저 연락하는 구조이며, Soli는 판단이 아니라 근거 요약을 담당한다 — 오늘 당장 문을 두드려야 할 한 사람을 찾아내는 것이 목표다. Soli 복지 매칭은 감지 이후 지원 연결을 돕는 보조 기능으로 제공되며, 실제 지원 집행 직전에는 수급자 본인 여부를 다시 확인한다.

### 3-1. 대상자 웹 화면 및 대상자 접점

#### Soli 정기 안부 확인 (1차 감지 채널)

신청주의 장벽으로 스스로 도움을 요청할 수 없는 대상자를 위해 Soli가 먼저 연락하는 구조다. 복지대상자와 응급 돌봄 수요가 빠르게 늘어나는 현장에서 담당자의 눈과 귀로서, 격월 행정 데이터가 놓치는 감정 변화와 응답 공백을 상시 포착한다.

| 항목 | 내용 |
|:---|:---|
| 대상 | 지자체가 이미 관리 중이거나 담당자가 현장에서 확인한 독거노인·취약계층 |
| 등록 주체 | 복지 담당자 또는 파일럿 운영자 |
| 방식 | 대상자 본인 동의 후 Soli가 매일 또는 격일로 짧은 안부 대화를 시작 |
| 감지 | 응답 공백, 부정 감정 표현, 위기 키워드 변화 감지 |
| 요약 방식 | 단순 위험 알림이 아니라 어떤 대화 맥락에서 신호가 감지되었는지 담당자에게 요약 — 빠른 의사결정 지원 |
| 알림 | 위험 신호가 누적되면 담당자 대시보드 우선순위에 자동 반영 |
| Phase 1 범위 | 외부 가족 초대·가족 알림 없이 담당자 관리 대상의 정기 안부 확인만 시연 |

#### Soli 복지 매칭

| 항목 | 내용 |
|:---|:---|
| 방식 | 사용자가 입력한 기본 조건(연령대·지역·가구 형태) + 선택형 상황 체크리스트를 함께 활용해 복지 후보 매칭 |
| 데이터 | 한국사회보장정보원 중앙부처·지자체 복지서비스 API 실연동 (공공데이터포털, 자동 승인, 무료) |
| 신청 연결 | 복지로(bokjiro.go.kr) 또는 정부24 외부 링크로 연결 |

**Phase 2 고려 사항**: 이웃·관리인이 "이 분이 걱정돼요" 수준의 낮은 진입장벽으로 안부를 공유하는 커뮤니티 케어 채널은 파일럿 이후 단계에서 추가 검토한다. MVP 범위에서는 제외.

### 3-2. AI 챗봇 Soli

| 기능 | 상세 |
|:---|:---|
| 정기 안부 대화 | 독거노인·취약계층에게 매일 또는 격일로 짧은 안부 질문 |
| 24/7 대화 접점 | 사용자가 원할 때 일상 대화 + 정서 지지 |
| 위기 감지 | 위기 키워드 감지 → 위기도 스코어 자동 산출 |
| Soli 복지 매칭 | 감지 이후 지원 연결 보조 — 대화 맥락·기본 조건·상황 체크리스트 기반 수급 가능 복지 후보 추천 |
| 첫 접점 역할 | 공무원 방문 전 신뢰 관계 형성 |

**위기도 스코어링 기준 (예정, 가중치)**:
- 대화 감정 패턴 — 부정 키워드 빈도, 절망 표현 (40%)
- 응답 공백·빈도 이상 — 평소 대비 응답 감소 (35%)
- 사회적 고립도 지표 (25%)

**공공마이데이터 위기 신호 보조 입력**: 담당자가 케이스 등록 시 체크리스트로 입력한 건보료 체납·실직·수급 이탈 등의 신호는 사회적 고립도 지표에 추가 반영한다. MVP에서는 담당자가 직접 입력하거나 Mock 데이터로 시연하며, 파일럿 이후 공공마이데이터 실연동으로 자동화한다.

### 3-3. 담당자 대시보드 (B2G)

| 기능 | 상세 |
|:---|:---|
| 우선순위 표시 | "오늘 확인 필요한 사람 N명" 자동 정렬 |
| 통합 뷰 | 위기 신호 수준 + Soli 대화 요약 + 응답 공백·위험 표현 근거 |
| 집행 연결 | 승인 버튼 → 지원 집행 연결 |

### 3-4. OmniOne Chain 기록

OmniOne Chain은 개인정보 원문 저장소가 아니라 VC 발급·승인·집행 이벤트가 특정 시점에 발생했음을 증명하는 사후 검증 레이어다. MVP 데모에서는 실제 주소·상세 상황 설명을 저장하지 않고, 파일럿 이후 실제 운영이 필요할 때만 DB 암호화 저장을 검토한다.

| 기록 항목 | 내용 |
|:---|:---|
| 지원 확인 VC 발급 이력 | VC 메타데이터 해시 + 발급 타임스탬프 |
| 판단 이력 | 위기도 스코어 해시 + 담당자 승인 기록 해시 |
| 집행 이력 | 지원 종류·금액 해시 + 수혜자 DID 해시 |

### 3-5. OmniOne 기술 역할 분담

| 기술 | 역할 | Phase 1 사용 지점 |
|:---|:---|:---|
| OmniOne CX | 해커톤 제공 SDK·가이드 기반 표준 인증창 연동으로 모바일 신분증 검증 결과 수신 | 서비스 최초 가입·로그인, 수급자 집행 직전 확인 |
| Open DID / VC | 인증된 행위 주체를 실명 대신 DID 해시와 VC 검증 결과로 연결 | 지원 확인 VC 발급, VP 검증 |
| OmniOne Chain | 이벤트 원문이 아니라 해시를 기록해 사후 검증 | 지원 확인 VC 메타데이터, 담당자 승인, 지원 집행 txId |

**데이터 경계:**
- 저장하지 않음: 모바일 신분증 원문, DID 원문
- MVP 데모 저장: DID 해시, 연락처 해시, 지역 코드, Soli 대화 암호문, 담당자 처리 상태
- 파일럿 이후 암호화 저장 검토: 주소, 상세 상황 설명 등 실제 운영에 필요한 민감정보
- 온체인 기록: DID 해시, 이벤트 해시, 타임스탬프, 지원 확인 VC 메타데이터 해시

### 3-6. 복지 데이터 연동 구조

#### 복지서비스 목록·추천 (실연동)

| 항목 | 내용 |
|:---|:---|
| 데이터 출처 | 한국사회보장정보원 — 중앙부처복지서비스 API, 지자체복지서비스 API |
| API | https://www.data.go.kr/data/15090532/openapi.do (중앙부처), https://www.data.go.kr/data/15108347/openapi.do (지자체) |
| 승인 | 공공데이터포털 개발·운영 자동 승인, 무료 |
| 제공 항목 | 서비스명, 서비스요약, 지원대상, 선정기준, 신청방법, 소관부처, 서비스URL |
| 매칭 입력 | 사용자가 입력한 기본 조건(연령대·지역·가구 형태), 선택형 상황 체크리스트(실직·질병·주거 불안·돌봄 공백 등) |
| Soli 활용 | 직접 입력한 기본 조건과 상황 체크리스트를 정규화 → API의 지원대상·선정기준 텍스트와 조건 필터링 → 해당 복지서비스 추천 |
| 신청 연결 | 서비스URL로 복지로(bokjiro.go.kr) 또는 정부24 외부 링크 |

#### 공공마이데이터 위기 신호 보조 데이터 (MVP Mock 시연, 파일럿 실연동)

공공마이데이터(행안부 178종, gov.kr/portal/mydata)는 개인이 본인 동의 후 행정·공공기관 보유 데이터를 제3자 서비스에 전송하는 구조다. NOA가 "이용기관"으로 등록되면 가입 시 사용자 동의를 받아 아래 항목을 수신할 수 있다.

**활용 가능 항목 및 위기 신호 의미**

| 제공기관 | 공공마이데이터 항목명 | 위기 신호 의미 | 활용도 |
|:---|:---|:---|:---|
| 근로복지공단 | 고용보험피보험자격이력내역서 | 취득·상실 이력으로 실직 시점 직접 확인 | 높음 |
| 국민건강보험공단 | 일반건강검진정보 | 장기 미수검 = 자기방치 신호 | 높음 |
| 보건복지부 | 국민기초생활수급자증명서 | 수급 이탈 시 복지 사각지대 재진입 | 높음 |
| 국민건강보험공단 | 4대사회보험료완납증명서 | 발급 불가 상태 = 건보료 체납 간접 확인 | 중간 |
| 국민연금공단 | 국민연금가입자가입증명 | 가입 기간 공백으로 납부 중단 확인 | 중간 |
| 국세청 | 소득금액증명 | 소득 급감 또는 0 확인 | 중간 |
| 근로복지공단 | 보험급여지급확인원 | 실업급여 수급 = 실직 후 생계 불안 | 중간 |
| 보건복지부 | 차상위계층확인서 | 경계선 취약계층 여부 | 중간 |
| 보건복지부 | 장애인증명서 | 돌봄 공백 위험 | 중간 |

**포함되지 않는 항목**: 수도 요금(지자체 별도 운영), 전기요금(KEPCO 별도 계약)은 공공마이데이터 178종 범위 밖이다.

**행복e음 참고**: 위 항목 대부분이 행안부 사회보장정보시스템(행복e음)에 이미 집약되어 있다. 담당자는 케이스 등록 시 행복e음에서 확인한 위기 신호를 체크리스트로 입력할 수 있으며, 이것이 MVP의 현실적 대안이다.

**MVP 시연 방법 (2가지 경로)**

경로 A — 담당자 체크리스트 입력 (즉시 구현 가능):
- 담당자 케이스 등록 화면에 위기 신호 체크리스트 추가 (고용보험 상실, 건강검진 미수검, 수급 이탈, 건보료 체납 등)
- 체크리스트 값이 위기도 스코어의 사회적 고립도 지표에 반영되는 흐름 시연

경로 B — Mock 공공마이데이터 연동 시연 (이용기관 심사 전 fallback):
- 위 항목과 동일한 구조의 Mock JSON 데이터를 사전 입력
- 사용자 가입 시 "공공마이데이터 제공 동의" 화면 → 데이터 수신 → 위기 신호 자동 반영 흐름을 Mock으로 시연

**실연동 조건**: 공공마이데이터 이용기관 승인(행안부 심사, 법인 필요) 후 파일럿 단계에서 자동 연동을 추진한다.

#### 개인 자격 확인 (파일럿 이후)

MVP에서는 개인 행정정보 조회를 구현하지 않는다. 공공 마이데이터 이용기관 승인(행안부 심사, 법인 필요)이 전제 조건이며, 파일럿 단계에서 지자체 또는 승인된 이용기관 구조로 연계를 추진한다.

## 5. MVP 범위

### 5-1. 5월 제안서 제출물 (2026-05-31 온라인 모집 마감)

제안서 단계에서는 구현물이 없다는 전제로, 문제 정의·해결 방식·차별성·사업성·MVP 구현 가능성을 명확히 증명한다.

**제출 대상:**
- 공모 제안서 PPTX (refs/TrackNo_팀명_프로젝트명_yymmdd.pptx 양식 기반)
- 제안 요약서 (최대 2p, 표 수정 불가)
- 제안내용의 목적 및 필요성
- 제안내용의 상세 설명
- 제안내용의 차별성
- 기대효과
- 관련 자료 및 기타 보충 자료

**미포함:**
- 실제 동작하는 웹앱, API, DB, OmniOne CX/Chain 연동 구현물

### 5-2. 예선 대면평가 준비물 (2026-06-24 ~ 2026-06-25)

예선에 선정될 경우, 8분 발표와 5분 Q&A 기준으로 제안서 내용을 발표 자료화한다.

**중점:**
- 창의성 35점: Soli AI 선제 감지 + 담당자 돌봄 연결 + 모바일 신분증 + 온체인 기록 조합
- 실현 가능성 35점: MVP 개발 단계 기간 내 구현 가능한 4개 핵심 플로우
- 사업성 30점: B2G SaaS, 보험사 공동 실증, 비식별 데이터 상품 수익 구조

### 5-3. MVP 구현 범위 (2026-07-01 ~ 2026-09-21)

**MVP로 구현해야 할 범위:**
- Soli 챗봇 (위기 키워드 감지 + 위기도 스코어 출력)
- Soli 복지 매칭 (지원 연결 보조: 한국사회보장정보원 API 실연동 + 복지로 외부 신청 링크)
- 담당자 대시보드 (위기 우선순위 리스트)
- OmniOne CX 모바일 신분증 연동 (가입·로그인, 집행 직전)
- Open DID 지원 확인 VC/VP 데모 + OmniOne Chain txId 표시 (가점 +10% 확보 필수)

**MVP 우선순위:**
- 가점 필수 데모: Soli 감지 → 담당자 승인 → 집행 직전 본인 확인 → 지원 확인 VC 발급 → VP 검증 → 지원 완료 증명 txId 표시
- 최소 성공 기준: VC 1개 발급, VP 1회 검증, OmniOne Chain txId 1개 화면 표시
- fallback: 외부 SDK 또는 네트워크 연동 지연 시 Mock 인증·데모 데이터·녹화 영상으로 동일한 흐름을 설명하되, 제출 목표는 최소 1개 실제 VC/VP/txId 성공

**미포함 (이후 단계):**
- 공공마이데이터 실연동 (이용기관 심사 필요 — MVP는 담당자 체크리스트 입력 + Mock 데이터 흐름 시연, 파일럿 단계에서 자동 연동 추진)
- 실제 행정 시스템 연동
- 보험사 대상 비식별 데이터 상품화·공동 실증 계약
- 재무 실데이터
- Phase 2 가족 안부 케어
- 보험사 데이터 API (수천 명 이상 축적 후 장기 검토)

|:---|:---|
