## 1. 개발 단계

**현재 상태**: 제안서 작성 단계 (구현 없음)
- 2026-05-31 온라인 모집 마감 전 공모 제안서 제출이 1차 목표
- 현재 문서는 제안서 본문, 발표 흐름, MVP 구현 계획을 정리하기 위한 설계 자료
- Next.js + TypeScript 기반 시민 앱 및 담당자 대시보드는 MVP 구현 예정
- AI 챗봇 Soli의 위기 키워드 감지·위기도 스코어링은 제안서 내 핵심 구현 계획으로 제시
- OmniOne CX (모바일 신분증) + Open DID + OmniOne Chain 연동은 MVP에서 구현할 선택조건 가점 전략
- 공공 마이데이터는 MVP에서 동의 화면과 표준 API 기반 Mock 자격 확인 결과로 시연하고, 실제 운영 API 계약·심사 완료는 파일럿 단계에서 추진

## 2. 기술 스택

| 레이어 | 기술 |
|:---|:---|
| Frontend | Next.js + TypeScript |
| Backend | NestJS + PostgreSQL |
| AI | Claude API (한국어 복지 상담 최적화) |
| Auth | OmniOne CX (모바일 신분증), Open DID |
| Public Data | 공공 마이데이터 동의·자격 확인 흐름 (MVP Mock 응답) |
| Blockchain | OmniOne Chain |
| Infra | Vercel + AWS |

## 3. 핵심 기능 명세

### 3-1. 시민 앱 및 대상자 접점

#### 이웃 제보

| 항목 | 내용 |
|:---|:---|
| 주요 제보 주체 | 고시원·쪽방 관리인, 생활지원사, 편의점 점주, 통장/반장 등 현장 접점. 일반 시민은 보조 채널 |
| 제보 동기 | 관리 리스크 (고독사 발생 시 특수청소 비용·악취·임대 기피 방지) — 관리인·건물주에게 실질적 이해관계 존재 |
| 입력 | 주소 + 상황 텍스트 |
| 신원 확인 | OmniOne CX 모바일 신분증 (제보자 본인 확인) |
| 목적 | 허위 신고 방지, 제보자 책임 귀속 |
| 출력 | 담당자 대시보드 위기 알림 생성 + 안심 제보 인증서 VC 발급 (제보 이력 OmniOne Chain 기록 → 반복 신원확인 없이 후속 확인 가능) |

#### 자가 복지 신청

| 항목 | 내용 |
|:---|:---|
| 방식 | Soli와 대화 → 받을 수 있는 복지 자동 매칭 |
| 자격 확인 | OmniOne CX 본인 확인 → 공공 마이데이터 동의 → Mock 자격 확인 결과 표시 |
| 신청 | 원스톱 신청 연결 |

#### 담당자 등록 기반 정기 안부 확인

| 항목 | 내용 |
|:---|:---|
| 대상 | 지자체가 이미 관리 중이거나 현장 제보로 확인된 독거노인·취약계층 |
| 등록 주체 | 복지 담당자 또는 파일럿 운영자 |
| 방식 | 대상자 본인 동의 후 Soli가 매일 또는 격일로 짧은 안부 대화를 시작 |
| 감지 | 응답 공백, 부정 감정 표현, 위기 키워드 변화 감지 |
| 알림 | 위험 신호가 누적되면 담당자 대시보드 우선순위에 반영 |
| Phase 1 범위 | 가족 초대·자녀 알림 없이 담당자 관리 대상의 정기 안부 확인만 시연 |

### 3-2. AI 챗봇 Soli

| 기능 | 상세 |
|:---|:---|
| 정기 안부 대화 | 독거노인·취약계층에게 매일 또는 격일로 짧은 안부 질문 |
| 24/7 대화 접점 | 사용자가 원할 때 일상 대화 + 정서 지지 |
| 위기 감지 | 위기 키워드 감지 → 위기도 스코어 자동 산출 |
| 복지 매칭 | 대화 맥락 기반 수급 가능 복지 자동 추천 |
| 첫 접점 역할 | 공무원 방문 전 신뢰 관계 형성 |

**위기도 스코어링 기준 (예정)**:
- 대화 감정 패턴 (부정 키워드 빈도, 절망 표현)
- 응답 빈도 이상 (평소 대비 응답 감소)
- 이웃 제보 수신 이력
- 사회적 고립도 지표

### 3-3. 담당자 대시보드 (B2G)

| 기능 | 상세 |
|:---|:---|
| 우선순위 표시 | "오늘 확인 필요한 사람 N명" 자동 정렬 |
| 통합 뷰 | 위기 스코어 + 제보 내역 + Soli 대화 요약 |
| 집행 연결 | 승인 버튼 → 지원 집행 연결 |

### 3-4. OmniOne Chain 기록

OmniOne Chain은 개인정보 원문 저장소가 아니라 제보·동의·승인·집행 이벤트가 특정 시점에 발생했음을 증명하는 사후 검증 레이어다. 주소, 상황 설명, 대화 원문, 마이데이터 원문은 온체인에 올리지 않고 DB에 암호화 저장한다.

| 기록 항목 | 내용 |
|:---|:---|
| 제보 이력 | 제보자 DID 해시 + 타임스탬프 + 상황 해시 |
| 판단 이력 | 위기도 스코어 해시 + 담당자 승인 기록 해시 |
| 집행 이력 | 지원 종류·금액 해시 + 수혜자 DID 해시 |
| 마이데이터 동의 이력 | 정보 제공 동의 완료 사실 해시 + Open DID Delegation 기록 (개인정보 원문 미포함) — 사후 검증 가능한 동의·확인 이력으로 활용 |
| CSR 기부 | 기업명 해시 + 집행 금액 집계 + 수혜자 통계 → ESG 리포트 자동 생성 |

### 3-5. OmniOne 기술 역할 분담

| 기술 | 역할 | Phase 1 사용 지점 |
|:---|:---|:---|
| OmniOne CX | 모바일 신분증 기반 본인 확인 | 제보자 인증, 대상자 마이데이터 동의 전 확인, 수급자 집행 직전 확인 |
| Open DID / VC | 인증된 행위 주체를 실명 대신 DID 해시와 VC 검증 결과로 연결 | 제보자 DID 발급, 인증 완료 배지, 제보 이력 VC |
| OmniOne Chain | 이벤트 원문이 아니라 해시를 기록해 사후 검증 | 제보 접수, 자격 확인 이벤트, 담당자 승인, 지원 집행 TX 해시 |

**데이터 경계:**
- 저장하지 않음: 모바일 신분증 원문, 마이데이터 원문, DID 원문
- DB 암호화 저장: 주소, 상황 설명, Soli 대화 전문, 담당자 처리 상태
- 온체인 기록: DID 해시, 이벤트 해시, 타임스탬프, 집행 증명 해시

### 3-6. 공공 마이데이터 자격 확인 흐름

| 항목 | 내용 |
|:---|:---|
| 목적 | 대상자 본인이 동의할 때만 사용. (1) 대상자가 Soli에서 자가 신청 시 복지 자격 확인 (2) 담당자 현장 방문 시 대상자가 동의하면 즉시 자격 보조 정보 확인 → 서류 보완 재방문 방지, 한 번 방문으로 상황 파악·지원 승인 동시에 처리 |
| MVP 시연 | 대상자 동의 화면, 표준 API 기반 Mock 조회 결과, 담당자 대시보드 확인 배지 |
| 조회 후보 | 건강보험 자격득실, 가구원 구성 등 자격 확인 보조 정보 |
| API 접근 경로 | 공공 마이데이터 포털 '본인정보 제공 요구' API 활용 — 대상자가 PDF 또는 데이터 형태로 본인 자격증명을 직접 받아 담당자에게 제출. MVP에서는 표준 API 기반 Mock 응답으로 시연 |
| 저장 원칙 | 원문 저장 금지, 자격 확인 결과와 이벤트 해시만 저장 |
| 범위 제한 | 최종 자격 판단은 담당자가 수행, 실제 운영 API 연동은 파일럿 단계 |

## 4. 필수조건 충족 (해커톤 Track 2)

### 모바일 신분증 활용 (필수조건)

| 사용 시점 | 역할 |
|:---|:---|
| 이웃 제보 시 | 제보자 신원 확인 → 허위 신고 방지 |
| 복지 자격 확인 시 | 대상자 본인 확인 → 마이데이터 동의 기반 자격 확인 보조 |
| 지원 집행 직전 | 수급자 본인 확인 → 부정 수급 방지 |

### 공공 마이데이터 활용

| 사용 시점 | 역할 |
|:---|:---|
| 복지 자격 확인 시 | 대상자 동의 기반 자격 확인 보조 정보 조회 |
| 담당자 검토 시 | 자격 확인 배지와 요약 결과 표시 |
| 기록 시 | 자격 확인 이벤트 해시만 저장, 원문 미저장 |

### Open DID + OmniOne Chain (선택조건 +10%)

| 기술 | 활용 |
|:---|:---|
| Open DID | 시민 제보자 DID 발급 → VC 기반 신원 증명 |
| OmniOne Chain | 제보·동의·승인·집행 이벤트의 해시 기록, CSR 기부 투명화 |

## 5. MVP 범위

### 5-1. 5월 제안서 제출물 (2026-05-31 온라인 모집 마감)

제안서 단계에서는 구현물이 없다는 전제로, 문제 정의·해결 방식·차별성·사업성·MVP 구현 가능성을 명확히 증명한다.

**제출 대상:**
- 공모 제안서 PPTX (refs/TrackNo_팀명_프로젝트명_yymmdd.pptx 양식 기반)
- 제안 요약서 (최대 2p, 표 수정 불가)
- 제안내용의 목적 및 필요성
- 제안내용의 상세 설명
- 제안내용의 차별성
- 기대효과
- 관련 자료 및 기타 보충 자료

**미포함:**
- 실제 동작하는 웹앱, API, DB, OmniOne CX/Chain 연동 구현물

### 5-2. 예선 대면평가 준비물 (2026-06-24 ~ 2026-06-25)

예선에 선정될 경우, 8분 발표와 5분 Q&A 기준으로 제안서 내용을 발표 자료화한다.

**중점:**
- 창의성 35점: 이웃 제보 + AI 근거 요약 + 모바일 신분증 + 온체인 기록 조합
- 실현 가능성 35점: MVP 개발 단계 기간 내 구현 가능한 4개 핵심 플로우
- 사업성 30점: B2G SaaS, 보험, CSR, 익명 집계 데이터 API 수익 구조

### 5-3. MVP 구현 범위 (2026-07-01 ~ 2026-09-21)

**MVP로 구현해야 할 범위:**
- Soli 챗봇 (위기 키워드 감지 + 위기도 스코어 출력)
- 이웃 제보 플로우 (OmniOne CX 모바일 신분증 실연동)
- 공공 마이데이터 자격 확인 흐름 (동의 화면 + Mock 조회 결과 + 담당자 확인 배지)
- 담당자 대시보드 (위기 우선순위 리스트)
- OmniOne Chain 집행 기록 실연동 (가점 +10%)

**미포함 (이후 단계):**
- 실제 행정 시스템 연동
- 실제 공공 마이데이터 운영 API 계약·심사 완료
- 보험사 API 연동
- 재무 실데이터
- Phase 2 가족 안부 케어

## 6. Non-Functional Requirements

| 항목 | 요구사항 |
|:---|:---|
| 개인정보 보호 | Soli 대화 데이터 암호화 저장, 담당자만 접근 가능 |
| 마이데이터 | MVP에서는 표준 API 기반 Mock 응답 사용, 원문 저장 금지, 보험회사 제공 제외 |

| 보안 | 모바일 신분증 연동 시 OmniOne CX 표준 프로토콜 준수 |
