## 3. 핵심 기능 명세

NOA의 핵심은 복지 신청을 기다리는 서비스가 아니라, AI 솔이가 먼저 전화를 걸어 위험 신호를 발견하는 선제 돌봄 인프라다. 신청주의 장벽을 넘을 수 없는 대상자에게 솔이가 먼저 닿고, 통화 신호·부재 패턴·위험 표현을 근거로 담당자가 오늘 우선 확인해야 할 사람을 보여준다. 솔이는 지원 여부를 판단하지 않는다. 판단은 담당자가 하고, 솔이는 "오늘 당장 문을 두드려야 할 한 사람"을 찾기 위한 근거 요약을 담당한다. NOA 사용자는 서비스 최초 가입·로그인 시 OmniOne CX 본인확인을 완료하며, 실제 지원 집행 직전에는 대상자 본인 여부를 다시 확인한다. 솔이 복지 매칭은 감지 이후 지원 연결을 돕는 보조 기능으로 제공한다.

### 3-1. 대상자 접점 및 솔이 정기 전화

#### 솔이 정기 전화 통화 (1차 감지 채널)

기존 복지는 신청한 사람을 처리한다. NOA는 신청하지 못한 사람에게 먼저 닿는다. 복지대상자와 응급 돌봄 수요가 빠르게 늘어나는 현장에서 솔이는 담당자의 눈과 귀로 작동하며, 격월 행정 데이터가 놓치는 감정 변화·목소리 신호·응답 공백을 정기 통화로 포착한다. 담당자 1명이 관리하는 수백 명에게 솔이가 주 1회 직접 전화하고, 위험 신호가 쌓이면 통화 주기를 높여 담당자의 확인 우선순위에 반영한다.

| 항목 | 내용 |
|:---|:---|
| 대상 | 55세 이상 누구나. 우선 대상: 독거노인·고립 위험군·기초연금·긴급복지 상담 경험자·담당자가 확인한 취약계층 |
| 등록 방식 | ① 담당자 직접 등록 (기존 관리 대상자) ② 현장 QR 자가 가입 (주민센터·복지관·경로당·보건소·아파트 관리사무소·병원·약국) |
| 낙인 완화 | "취약계층 등록"이 아니라 "55세 이상 안부 연결 신청"으로 프레이밍 — 진입 장벽 최소화 |
| 방식 | 대상자 본인 동의 후 솔이가 등록된 전화번호로 아웃바운드 전화 발신 — 약 5분 안부 대화 |
| 통화 구조 | ① 인사·이름 호명 (30초) ② 자유 안부 대화 (3분) ③ 확인 질문 1–2개 (1분) ④ 마무리·다음 예고 (30초) |
| 전화 주기 | 위기 레벨에 따라 자동 조절 |
| 감지 | 목소리 에너지·감정 변화, 위기 키워드, 응답 공백, 부재 패턴 |
| 요약 방식 | 통화 종료 후 STT 변환 → Claude API 분석 → 담당자에게 AI 요약 전달 (원문 아님) |
| 알림 | 위험 신호 누적 또는 부재 반복 시 담당자 대시보드 우선순위 자동 반영 |
| Phase 1 범위 | 담당자 직접 등록 + 현장 QR 자가 가입 2가지 경로 시연. 외부 가족 초대·가족 알림 없음 |

**전화 주기 — 위기 레벨 연동**

| 위기 레벨 | 위기 점수 | 솔이 전화 주기 |
|:---|:---|:---|
| 정상 | 0–3 | 주 1회 |
| 주의 | 4–6 | 주 2회 |
| 경고 | 7–8 | 주 3–4회 + 담당자 알림 |
| 위급 | 9–10 | 즉시 담당자 알림 + 긴급 연락 |

**부재 처리**

| 상황 | 처리 |
|:---|:---|
| 1회 부재 | 2시간 후 재시도 |
| 2회 연속 부재 | 담당자 알림 발송 |
| 3회 연속 부재 | 담당자 긴급 알림 + 사전 동의된 비상 연락처 확인 |

### 3-2. 솔이 AI 전화 통화

솔이는 CLOVA Voice 기반 따뜻한 한국어 여성 음성으로, 등록된 대상자에게 아웃바운드 전화를 발신한다. 솔이 전화는 단순 안부 확인이 아니라, 도움을 요청하지 못한 사람에게 먼저 닿아 담당자의 현장 확인을 앞당기는 1차 감지 채널이다.

NOA의 대상자 관리는 4단계 구조로 작동한다. 1단계는 모든 대상자 공통 AI 전화, 2단계는 SMS 또는 웹 채팅 중 사용자가 편한 채널을 선택하는 병렬 보조 대화, 3단계는 대시보드 통합 분류, 4단계는 담당자 개입이다. 솔이는 복지 판단을 하지 않는다 — 계속 안부를 묻고 정리해서 담당자의 개입 타이밍을 앞당기는 역할이다.

**MVP 플랫폼**: 웹(Next.js) 단일 배포. 네이티브 앱은 해커톤 이후 단계에서 개발한다. SMS와 전화는 NOA 서버가 오케스트레이션하므로 사용자가 앱을 설치하지 않아도 1~2단계 서비스가 작동한다.

| 기능 | 상세 |
|:---|:---|
| 정기 전화 통화 | 위기 레벨에 따라 주 1–4회 아웃바운드 발신, 1회 약 5분 |
| 통화 구조 | 인사·이름 호명 → 자유 안부 대화 → 확인 질문 1–2개 → 마무리 |
| 음성 신호 감지 | 통화 중/통화 후 목소리 에너지·감정 변화·위기 키워드·대화 단절 신호 분석 |
| 부재 처리 | 2시간 후 재시도 → 2회 연속 부재 시 담당자 알림 → 3회 연속 담당자 긴급 알림 |
| SMS 관계 유지 | 전화 직후 또는 부재 후 자동 발송 — "말로 하기 어려운 내용은 문자로 남겨주세요". 답장 내용은 NOA 서버가 수신해 대시보드 요약에 반영 |
| 웹 채팅 선택 | 복지 매칭이 필요한 시점 또는 신뢰 형성 후 링크 발송 → /checkin·/welfare 진입. SMS와 병렬 운영하며 사용자가 편한 채널을 선택 |
| 통화 후 처리 | STT 변환 → Claude API 위기 분석 → 담당자 AI 요약 생성 |
| 솔이 복지 매칭 | 감지 이후 지원 연결 보조 — 통화 맥락·기본 조건·상황 체크리스트 기반 복지 후보 추천 |
| 첫 접점 역할 | 공무원 방문 전 신뢰 관계 형성, 피처폰 포함 모든 전화기 대응 |

**4단계 관리 구조**

NOA의 관리 구조는 4단계다. 1단계는 모든 대상자에게 공통으로 적용되는 솔이 AI 전화다. 대상자는 앱 설치 없이 전화를 받기만 하면 된다. 2단계는 SMS 또는 웹 채팅 중 사용자가 편한 채널을 선택하는 보조 대화 단계다. 두 채널은 병렬로 운영되며, 모든 신호는 3단계 담당자 대시보드에서 통합 분류된다. 4단계에서는 담당자가 요약 근거를 보고 전화·방문·복지 연결·지원 집행을 결정한다.

| 단계 | 채널 | 목적 | 솔이/NOA 역할 |
|:---|:---|:---|:---|
| 1단계 | AI 전화 | 접점 유지·안부 확인·위험 표현 감지·부재 패턴 확인 | 1차 안부 담당자 / 신호 감지자 |
| 2단계 | SMS 또는 웹 채팅 (사용자 선택) | 관계 유지·추가 정보 수집·복지 매칭 | SMS: 답장 수신 → 요약 반영 / 웹 채팅: 체크리스트·복지 후보 추천 |
| 3단계 | 대시보드 분류 | 위험 신호 수준별 담당자 우선순위 자동 분류 | 요약 생성·위기 스코어 산출 |
| 4단계 | 담당자 개입 | 경고 이상 케이스만 담당자가 직접 움직임 | 근거 요약 제공, 판단은 담당자가 |

2단계는 강제 전환이 아니다. 전화 수신 후 SMS가 편한 사용자는 문자로, 웹이 익숙한 사용자는 웹 채팅으로 계속 소통한다. 두 채널 모두 NOA 서버가 수신·처리하며 대시보드에 반영된다.

**3단계 분류 기준**

| 상태 | 처리 |
|:---|:---|
| 정상 | 다음 정기 전화 예약 |
| 약한 걱정 | SMS 또는 웹 체크인 유도, 전화 주기 소폭 증가 |
| 주의 | SMS·웹 체크리스트 유도, 대시보드 표시 |
| 경고 | 담당자 확인 요청 |
| 위급 | 담당자 긴급 알림 + 사전 동의 비상 연락처 확인 |

**위기도 스코어링 기준 (예정, 가중치)**:
- 통화 음성 감정 패턴 — 부정 키워드 빈도, 절망 표현, 목소리 에너지 저하 (40%)
- 부재·통화 빈도 이상 — 전화 미수신, 조기 종료, 평소 대비 반응 감소 (35%)
- 사회적 고립도 지표 (25%)

**공공마이데이터 위기 신호 보조 입력**: 담당자가 케이스 등록 시 체크리스트로 입력한 건보료 체납·실직·수급 이탈 등의 신호는 사회적 고립도 지표에 추가 반영한다. MVP에서는 담당자가 직접 입력하거나 Mock 데이터로 시연하며, 파일럿 이후 공공마이데이터 실연동으로 자동화한다.

### 3-3. 담당자 대시보드 (B2G)

| 기능 | 상세 |
|:---|:---|
| 우선순위 표시 | "오늘 확인 필요한 사람 N명" 자동 정렬 |
| 통합 뷰 | 위기 신호 수준 + 솔이 통화 AI 요약 + 부재 패턴·위험 표현 근거 |
| 집행 연결 | 승인 버튼 → 지원 집행 연결 |

### 3-4. OmniOne Chain 기록

OmniOne Chain은 개인정보 원문 저장소가 아니라 VC 발급·승인·집행 이벤트가 특정 시점에 발생했음을 증명하는 사후 검증 레이어다. MVP 데모에서는 실제 주소·상세 상황 설명을 저장하지 않고, 파일럿 이후 실제 운영이 필요할 때만 DB 암호화 저장을 검토한다.

| 기록 항목 | 내용 |
|:---|:---|
| 지원 확인 VC 발급 이력 | VC 메타데이터 해시 + 발급 타임스탬프 |
| 판단 이력 | 위기도 스코어 해시 + 담당자 승인 기록 해시 |
| 집행 이력 | 지원 종류·금액 해시 + 수혜자 DID 해시 |

### 3-5. OmniOne 기술 역할 분담

| 기술 | 역할 | Phase 1 사용 지점 |
|:---|:---|:---|
| OmniOne CX | 해커톤 제공 SDK·가이드 기반 표준 인증창 연동으로 모바일 신분증 검증 결과 수신 | 서비스 최초 가입·로그인, 대상자 집행 직전 확인 |
| Open DID / VC | 인증된 행위 주체를 실명 대신 DID 해시와 VC 검증 결과로 연결 | 지원 확인 VC 발급, VP 검증 |
| OmniOne Chain | 이벤트 원문이 아니라 해시를 기록해 사후 검증 | 지원 확인 VC 메타데이터, 담당자 승인, 지원 집행 txId |

**데이터 경계:**
- 저장하지 않음: 모바일 신분증 원문, DID 원문
- MVP 데모 저장: DID 해시, 연락처 해시, 지역 코드, 솔이 통화 STT 암호문, 담당자 처리 상태
- 파일럿 이후 암호화 저장 검토: 주소, 상세 상황 설명 등 실제 운영에 필요한 민감정보
- 온체인 기록: DID 해시, 이벤트 해시, 타임스탬프, 지원 확인 VC 메타데이터 해시

### 3-6. 복지 데이터 연동 구조

#### 복지서비스 목록·추천 (실연동)

| 항목 | 내용 |

| 데이터 출처 | 한국사회보장정보원 — 중앙부처복지서비스 API, 지자체복지서비스 API |

| 제공 항목 | 서비스명, 서비스요약, 지원대상, 선정기준, 신청방법, 소관부처, 서비스URL |
| 매칭 입력 | 사용자가 입력한 기본 조건(연령대·지역·가구 형태), 선택형 상황 체크리스트(실직·질병·주거 불안·돌봄 공백 등) |

| 신청 연결 | 서비스URL로 복지로(bokjiro.go.kr) 또는 정부24 외부 링크 |

#### 공공마이데이터 위기 신호 보조 데이터 (MVP Mock 시연, 파일럿 실연동)

> **MVP 연동 구조 요약**
> - 실연동: 한국사회보장정보원 복지서비스 API (자동승인, 즉시 가능)
> - Mock: 공공마이데이터 개인 행정정보 (이용기관 승인 전 동일 구조로 흐름 검증)
> - 담당자 입력: 행복e음·현장 확인 기반 체크리스트

공공마이데이터(행안부, gov.kr/portal/mydata, 묶음정보 서비스 지속 확대 중)는 개인이 본인 동의 후 행정·공공기관 보유 데이터를 제3자 서비스에 전송하는 구조다. NOA가 "이용기관"으로 등록되면 가입 시 사용자 동의를 받아 아래 항목을 수신할 수 있다.

> **이용기관 신청 절차**: 이용기관 신청 → 현장실사 → 심사 → 승인(신청 후 3개월 이내 통보). 해커톤 MVP 기간에 신규 팀이 실데이터를 수신하는 구조는 현실적으로 어렵다. MVP는 Mock, 파일럿 이후 실연동이 맞는 방향이다.

**활용 가능 항목 및 위기 신호 의미**

| 제공기관 | 공공마이데이터 항목명 | 위기 신호 의미 | 활용도 |
|:---|:---|:---|:---|
| 근로복지공단 | 고용보험피보험자격이력내역서 | 취득·상실 이력으로 실직 시점 직접 확인 | 높음 |
| 국민건강보험공단 | 일반건강검진정보 | 장기 미수검 = 자기방치 신호 | 높음 |
| 보건복지부 | 국민기초생활대상자증명서 | 수급 이탈 시 복지 사각지대 재진입 | 높음 |
| 국민건강보험공단 | 4대사회보험료완납증명서 | 발급 불가 상태 = 건보료 체납 간접 확인 | 중간 |
| 국민연금공단 | 국민연금가입자가입증명 | 가입 기간 공백으로 납부 중단 확인 | 중간 |
| 국세청 | 소득금액증명 | 소득 급감 또는 0 확인 | 중간 |
| 근로복지공단 | 보험급여지급확인원 | 실업급여 수급 = 실직 후 생계 불안 | 중간 |
| 보건복지부 | 차상위계층확인서 | 경계선 취약계층 여부 | 중간 |
| 보건복지부 | 장애인증명서 | 돌봄 공백 위험 | 중간 |

**활용 항목 정리**: 건강보험·고용·소득·복지 자격 관련 증명·이력 정보를 기반으로 생활위기 가능성을 보조 판단한다. 수도·전기·가스·통신비 체납은 공공마이데이터 범위 밖(지자체·KEPCO 별도 계약)이며, 사용자 제출 증빙 또는 기관 제휴 이후 확장 범위로 분리한다.

**행복e음 참고**: 위 항목 대부분이 행안부 사회보장정보시스템(행복e음)에 이미 집약되어 있다. 담당자는 케이스 등록 시 행복e음에서 확인한 위기 신호를 체크리스트로 입력할 수 있으며, 이것이 MVP의 현실적 대안이다.

**MVP 시연 방법 (2가지 경로)**

경로 A — 담당자 체크리스트 입력 (즉시 구현 가능):
- 담당자 케이스 등록 화면에 위기 신호 체크리스트 추가 (고용보험 상실, 건강검진 미수검, 수급 이탈, 건보료 체납 등)
- 체크리스트 값이 위기도 스코어의 사회적 고립도 지표에 반영되는 흐름 시연

경로 B — Mock 공공마이데이터 연동 시연 (이용기관 심사 전 fallback):
- 위 항목과 동일한 구조의 Mock JSON 데이터를 사전 입력
- 사용자 가입 시 "공공마이데이터 제공 동의" 화면 → 데이터 수신 → 위기 신호 자동 반영 흐름을 Mock으로 시연

**실연동 조건**: 공공마이데이터 이용기관 승인(행안부 심사, 법인 필요) 후 파일럿 단계에서 자동 연동을 추진한다.

#### 개인 자격 확인 (파일럿 이후)

MVP에서는 개인 행정정보 조회를 구현하지 않는다. 공공 마이데이터 이용기관 승인(행안부 심사, 법인 필요)이 전제 조건이며, 파일럿 단계에서 지자체 또는 승인된 이용기관 구조로 연계를 추진한다.

## 5. MVP 범위

### 5-1. 5월 제안서 제출물 (2026-05-31 온라인 모집 마감)

제안서 단계에서는 구현물이 없다는 전제로, 문제 정의·해결 방식·차별성·사업성·MVP 구현 가능성을 명확히 증명한다.

**제출 대상:**
- 공모 제안서 PPTX (refs/TrackNo_팀명_프로젝트명_yymmdd.pptx 양식 기반)
- 제안 요약서 (최대 2p, 표 수정 불가)
- 제안내용의 목적 및 필요성
- 제안내용의 상세 설명
- 제안내용의 차별성
- 기대효과
- 관련 자료 및 기타 보충 자료

**미포함:**
- 실제 동작하는 웹앱, API, DB, OmniOne CX/Chain 연동 구현물

### 5-2. 예선 대면평가 준비물 (2026-06-24 ~ 2026-06-25)

예선에 선정될 경우, 8분 발표와 5분 Q&A 기준으로 제안서 내용을 발표 자료화한다.

**중점:**
- 창의성 35점: 솔이 AI 정기 전화 선제 감지 + 담당자 돌봄 연결 + 모바일 신분증 + 온체인 기록 조합
- 실현 가능성 35점: MVP 개발 단계 기간 내 구현 가능한 4개 핵심 플로우
- 사업성 30점: B2G SaaS, 보험사 공동 실증, 비식별 데이터 상품 수익 구조

### 5-3. MVP 구현 범위 (2026-07-01 ~ 2026-09-21)

**MVP로 구현해야 할 범위:**
- 솔이 AI 전화 통화 (아웃바운드 발신 + STT + GPT-4o mini 대화 + CLOVA Voice + SMS·웹 채팅 선택 채널 + 위기도 스코어 출력)
- 솔이 복지 매칭 (지원 연결 보조: 한국사회보장정보원 API 실연동 + 복지로 외부 신청 링크)
- 담당자 대시보드 (위기 우선순위 리스트 + 솔이 통화 AI 요약)
- OmniOne CX 모바일 신분증 연동 (가입·로그인, 집행 직전)
- Open DID 지원 확인 VC/VP 데모 + OmniOne Chain txId 표시 (가점 +10% 확보 필수)

**MVP 우선순위:**
- 가점 필수 데모: 솔이 전화 감지 → 담당자 승인 → 집행 직전 본인 확인 → 지원 확인 VC 발급 → VP 검증 → 지원 완료 증명 txId 표시
- 최소 성공 기준: VC 1개 발급, VP 1회 검증, OmniOne Chain txId 1개 화면 표시
- fallback: 외부 SDK 또는 네트워크 연동 지연 시 Mock 인증·데모 데이터·녹화 영상으로 동일한 흐름을 설명하되, 제출 목표는 최소 1개 실제 VC/VP/txId 성공

**미포함 (이후 단계):**
- 공공마이데이터 실연동 (이용기관 심사 필요 — MVP는 담당자 체크리스트 입력 + Mock 데이터 흐름 시연, 파일럿 단계에서 자동 연동 추진)
- 실제 행정 시스템 연동
- 보험사 대상 비식별 데이터 상품화·공동 실증 계약
- 재무 실데이터
- Phase 2 가족 안부 케어
- 보험사 데이터 API (수천 명 이상 축적 후 장기 검토)

## 6. Non-Functional Requirements

| 항목 | 요구사항 |
|:---|:---|
| 개인정보 보호 | 솔이 통화 STT 및 SMS 응답 데이터 암호화 저장, 담당자만 요약 접근 가능 |
| 복지 데이터 | 한국사회보장정보원 API 실연동, 개인 행정정보 조회 없음 |
| 온체인 데이터 | 집계·익명화 처리 후 기록 (개인 특정 불가) |
| 가용성 | 솔이 전화 스케줄링 24/7 가동 (Vercel Cron Jobs + 알리고 API) |
| 보안 | 모바일 신분증 연동 시 OmniOne CX 표준 프로토콜 준수 |
| 접근성 | 본인확인 시 OmniOne CX 사용. 이후 일상 이용(전화 수신·SMS 답장)은 WiFi·인터넷 없이 작동. 블록체인·대시보드는 담당자 측 인프라에서 처리 |

### 역할별 인터넷 의존도

| 역할 | 사용 채널 | WiFi·인터넷 필요 여부 |
|:---|:---|:---|
| 대상자 (어르신) | 솔이 전화 수신 + SMS 답장 | 불필요 — 2G/3G 포함 모든 통신망, 피처폰 가능 |
| 담당자 (공무원) | 대시보드 (/officer) | 필요 — 사무실 환경 |
| 블록체인 기록 확인 | /proof/:txId | 필요 — 담당자·감사관 사용 |
| 본인확인 | OmniOne CX | 필요 — 가입·집행 시점 |
